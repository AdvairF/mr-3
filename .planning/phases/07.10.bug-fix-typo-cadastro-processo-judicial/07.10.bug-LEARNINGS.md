## Lessons (3 lições centrais)

### 1. Bug pré-existente apareceu 3x antes de ser fixado — pattern observado
**Contexto:** typo `[object Object]` descoberto durante UAT 7.10a (2026-04-25), bloqueou UAT cross-feature 7.13 (2026-04-26), forçou SKIP de SC5+SC22 da 7.14 (2026-04-27). 3 phases consecutivas afetadas.
**Lição:** bugs pré-existentes em features dependentes (ProcessosJudiciais cadastro) **devem ser priorizados ALTA** assim que afetarem UAT da 2ª phase. Esperar pra "talvez phase futura" custa rounds e SCs SKIP.
**Onde aplicar:** durante UAT de phase N, se descobrir bug pré-existente bloqueando validação visual, bumpar severidade pra ALTA no backlog **mesmo que escopo seja trivial**. Phase trivial bloqueando 3 phases vale prioridade alta.

### 2. Investigação grep antes de planning revela escopo real
**Contexto:** briefing original tinha causa raiz como "concat objeto com string" (`dbInsert("processos_judiciais" + algo)`). Investigação grep revelou causa diferente: 4º arg objeto coerce-to-string em template literal. Bug em pattern oposto ao briefing.
**Lição:** grep audit pré-CONTEXT.md valida hipótese do briefing. Se hipótese errada, plan vai pro caminho errado (procurar concat onde não há, perder tempo).
**Onde aplicar:** discuss-phase de bug fixes deve incluir grep audit explícito como pre-condition do CONTEXT.md (já estava no protocolo desta phase, mas reforço o pattern).

### 3. Equivalência HTTP via shield 4 protege regressão silenciosa
**Contexto:** fix removeu `{ headers: { Prefer: "return=representation" } }`. Sem Shield 4 verificando que `Prefer:` ainda é hardcoded em `sb()` L25, fix poderia ter quebrado retorno do POST silenciosamente (PostgREST sem `Prefer: return=representation` retorna 201 sem body).
**Lição:** shields de equivalência HTTP/API são tão importantes quanto shields de motor. Quando remover código que parece redundante, sempre verificar onde a função "duplicada" vem (default, hardcoded, etc).
**Onde aplicar:** qualquer fix que remova arg/option de uma função, adicionar shield verificando que o behavior alvo ainda existe via outro path.

## Patterns (1 pattern reutilizável)

### Pattern: Shield 2 D-pre-11 anti-leakage com regex anchored em phases sequenciais
**Contexto:** Phase 7.14 D-pre-11 forensic exception em `processosJudiciais.js:71` (JOIN aninhado SEM filtro `deleted_at`). Phase 7.10.bug toca **outras linhas do mesmo arquivo** (L40 + L62), risco real de regressão acidental no L71.
**Estrutura:** shield regex anchored `devedor:devedores\([^)]*\).*deleted_at=is\.null` verifica que JOIN string em si permanece sem filtro, **enquanto comentário audit trail D-pre-11 (que menciona literal `deleted_at=is.null`) é preservado**.
**Aplicado em:** Plan 07.10.bug-01 Task 01-04 Shield 2. Reutilizável em qualquer phase que toque arquivo com exceções intencionais documentadas inline (forensic, legacy, security).

## Surprises (2 surpresas durante execução)

### Surprise 1: Causa raiz diferente do briefing
**Discovery:** briefing presumia concat string-objeto (`"processos_judiciais" + obj`). Investigação grep revelou causa real: 4º arg objeto em `sb()` coerce-to-string em template literal `${path}${extra}`. Pattern arquiteturalmente diferente.
**Resolução:** CONTEXT.md documentou causa raiz real com snippet de `sb()` signature + pattern do bug. Fix mudou de "achar string concat" pra "remover 4º arg redundante".

### Surprise 2: Bonus side-effect — desbloqueia SC5+SC22 da 7.14
**Discovery:** durante investigação 7.10.bug, percebi que mesmo `criar()` quebrado também afetava `adicionarDevedor()` (L62 mesmo bug). Fix de 2 linhas resolve cadastro processo + adição réu. Ambos eram pré-condições pra SC5 (badge forense polo passivo) + SC22 (filter lookup criação NOVA) da Phase 7.14 SHIPPED com SKIP.
**Resolução:** SUMMARY/LEARNINGS desta phase registra desbloqueio explícito. SC5+SC22 da 7.14 podem ser validados em UAT futuro sem necessidade de phase nova.
