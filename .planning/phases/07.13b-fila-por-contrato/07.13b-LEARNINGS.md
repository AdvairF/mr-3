## Decisions (12 D-pre originais + 3 D-pre POST-HOC)

### D-pre-1 — Fila puxa CONTRATOS (`contratos_dividas`), não devedores
**Why:** Refactor estrutural pedido pelo usuário — modelo "1 atendimento por contrato" alinha com forma como advogado opera (cada contrato tem ciclo próprio, evento próprio, status próprio). Devedor com 3 contratos não é "1 caso" — são 3 frentes de cobrança independentes.
**How applied:** `listarContratosParaFila` (rename de `listarDevedoresParaFila`) consome `contratos_dividas` com enrichment `_devedores`/`_credores`/`_pagamentos_divida`/`_dividas`. Toda função em `filaDevedor.js` operates contract-first.

### D-pre-2 — Cada contrato vira 1 linha (devedor com 2 contratos = 2 linhas)
**Why:** Refletir realidade: cada contrato precisa atendimento independente. Repetição visual aceita por simplicidade — banco minúsculo (5 devedores ativos), não há explosão combinatória.
**How applied:** UI `FilaPainel` lista contratos diretamente; busca por nome de devedor com 2 contratos retorna 2 linhas (Q7 confirmado em UAT-2 cross-entity).

### D-pre-3 — Por linha: nome devedor + credor + saldo atualizado + clique abre `DetalheContrato`
**Why:** Reusar componente `DetalheContrato.jsx` existente — view já valida saldos, encargos, parcelas, custas. Sem custo de UI nova.
**How applied:** Cada linha da `FilaPainel` clicável; handler navega pra `DetalheContrato` com `contrato.id`. Componente intocado cumulative.

### D-pre-4 — Status PER CONTRATO (7 valores)
**Why:** Status por devedor (legado) confunde quando devedor tem múltiplos contratos em fases diferentes (ex: contrato A em negociação, contrato B já notificado).
**How applied:** `STATUS_CONTRATO` const exportado em `constants.js` (7 valores: `em_cobranca`, `em_localizacao`, `em_negociacao`, `notificado`, `ajuizado`, `quitado`, `arquivado`). `STATUS_CONTRATO_TERMINAIS` (3 valores) + `STATUS_CONTRATO_ATIVOS` (4 valores) auxiliares. Migration 004 ALTER `contratos_dividas ADD COLUMN status TEXT NOT NULL CHECK (7 valores) DEFAULT 'em_cobranca'`.

### D-pre-5 — Telefone/CPF do devedor pode aparecer 2x (B1)
**Why:** Repetição é informação válida quando devedor tem múltiplos contratos. Sem complexidade extra de "agrupamento visual" — UAT-2 cross-entity confirmou que busca retorna N linhas para N contratos do mesmo devedor.
**How applied:** Render direto sem dedupe artificial.

### D-pre-6 — Toggle "Esconder quitados/arquivados" default ON
**Why:** Operador não quer ver contratos terminais por padrão (poluição visual). Mas UAT-4 valida toggle OFF mostrando reentrada/saída da Fila — útil pra auditoria.
**How applied:** Filtro client-side em `FilaPainel`/`FilaPesquisa`: `status NOT IN ('quitado','arquivado')` quando toggle ON. UAT-4 PASS.

### D-pre-7 — DROP legado de `eventos_andamento` (γ)
**Why:** Histórico de eventos legado vinculado a `devedor_id` é descartável (banco minúsculo, 5 devedores ativos). Atendimentos novos vão pro contrato. Migration TRUNCATE STRATEGY A aplicada (banco sem dados críticos pré-7.13b).
**How applied:** Migration 004 Bloco 3 `TRUNCATE eventos_andamento` (STRATEGY A). Bloco 4 `ALTER COLUMN contrato_id SET NOT NULL` posterior. UAT-3 confirmou tabela vazia pré-uso e exatamente N rows com `contrato_id` NOT NULL pós-uso.

### D-pre-8 — Bug fix incluso no rewrite (`contratos`→`contratos_dividas`)
**Why:** `filaDevedor.js` apontava para tabela `contratos` (inexistente — nome real é `contratos_dividas`) em 5+ callsites (`entrarNaFila` L253, `reciclarContratos` L491+L496, `listarFila` L585, etc). Funções dormentes em prod (Fila estava 100% quebrada via essas funções, mas listing usava outra rota). Fix-forward natural pelo rewrite.
**How applied:** Rewrite total `filaDevedor.js` no Plan 01 fixou 5+ stale refs. Sem isolar como sub-phase 7.13c — bug se autocorrigiu pelo refactor estrutural.

### D-pre-9 — Filtro `devedores.deleted_at=is.null` em `listarFila`
**Why:** Consistência com 7.14/D-pre-9 strategy b. Devedor soft-deleted não pode aparecer em Fila operacional.
**How applied:** `listarFila` aplica filtro `deleted_at=is.null` no fetch + drop client-side de items sem `_devedor` map. UAT-5 confirmou contrato some da Fila quando devedor é soft-deleted; reaparece quando `deleted_at=NULL` revertido.

### D-pre-10 — Migration manual robusta a 2 strategies (TRUNCATE preferida, DELETE fallback)
**Why:** Banco minúsculo permite TRUNCATE rápido. Mas RLS pode bloquear TRUNCATE em alguns ambientes (privilege requirement). DELETE respeita RLS — fallback documentado.
**How applied:** Migration 004 com 5 blocos sequenciais. STRATEGY A (TRUNCATE) aplicada em prod 2026-04-27 sem erro. STRATEGY B (DELETE) documentada no SUMMARY como fallback transferível para futuras γ-decisions.

### D-pre-11 — Side-effects fora do escopo preservados intactos

**How applied:** Zero modificações em módulos fora-Fila cumulative. Detalhe-Devedor preservado verbatim.

### D-pre-12 — Helper saldo por contrato: reusar `calcularDetalheEncargosContrato` (D-01 motor INTOCADO)
**Why:** Helper já existia em Phase 7.8 (D-02 adapter contrato-level que envolve motor `calcularDetalheEncargos` D-01 INTOCADO). Criar helper novo seria duplicação. Pattern transferível: phases que precisam de saldo por contrato reusam adapter existente.
**How applied:** Wrapper opcional `calcularSaldoContratoAtualizado` em `devedorCalc.js` (1-linha, delega ao adapter). Plan 01 introduziu wrapper sem modificar motor; Plan 02 UI consome `_pagamentos_divida` enriched no shape contrato.

## Lessons (10 lições centrais transferíveis)

### 1. Plan acceptance gates count keys distintas, não regex (PAUSA #1)
**Contexto:** Plan 01 PAUSA #1: regex `>=4` não bateu com 3 keys distintas reais (`_devedores`/`_credores`/`_pagamentos_divida`). Semântica B2.1 satisfeita, não code drift — gate measurement error.
**Lição:** Counts numéricos em gates devem bater com number de keys distintas reais. Auditar regex vs intenção semântica antes de checker round.
**Onde aplicar:** Qualquer gate `count >= N` em PLAN.md acceptance criteria.

### 2. Guard tests devem migrar explicitamente em rewrites (PAUSA #2-EXTENDED)
**Contexto:** Plan 01 PAUSA #2-EXTENDED: 4 tests classificados como guard/invariant defense (CR-01 SQL injection guard + CR-02 lock otimista race) não migraram automaticamente por "cobertura runtime intacta". Re-migração explícita salvou +13 asserts. CRUD trivial (TEST 5 `removerDaFila`) perdida — aceitável.
**Lição:** Tests defensivos contra refactor futuro silencioso (SQL injection, race conditions, semantic drift) NÃO migram por cobertura runtime. Exigem migração explícita em rewrites totais.
**Onde aplicar:** Qualquer rewrite total de `services/X.js` + `services/X.test.js` — auditar tests por classificação antes de copiar para schema novo.

### 3. Re-migração de tests adapt semantics, não copia verbatim (PAUSA #2-EXTENDED)
**Contexto:** Smoke `proximoContrato("' OR 1=1 --")` — HEAD esperava `success===false`. Função renamed/refactored adaptou para "não crasha + bool". +1 assert extra justificado pela semântica adapted.
**Lição:** Re-migração ajusta asserts à semântica REAL da função renamed/refactored, não copia comportamento histórico.
**Onde aplicar:** Qualquer port de tests entre schemas — identificar "comportamento histórico" vs "comportamento adapted" caso a caso.

### 4. Handler reaproveitando service legacy = bug semântico (Plan 02 Opção A)
**Contexto:** Botão "Atender" no rewrite UI Plan 02 chamava `proximoContrato()` sem ID — service retornava "fila vazia" mesmo com contratos visíveis. HEAD pre-7.13b chamava handler com ID explícito.
**Lição:** Handler UI reaproveitando service legacy precisa auditoria de signature alignment. Rewrite UI exige explicit pass dos IDs que o service espera — assumir "service legacy faz a coisa certa" é bug latente.
**Onde aplicar:** Rewrites UI que consomem services legacy ou refactored — auditar signature de cada handler em PAUSA pré-merge.

### 5. `ADD COLUMN IF NOT EXISTS` skip REFERENCES — FK órfã silenciosa (D-pre-13 + Migration 005)
**Contexto:** Migration 004 `ALTER TABLE eventos_andamento ADD COLUMN IF NOT EXISTS contrato_id UUID REFERENCES contratos_dividas(id)` — quando coluna já existia de tentativa anterior, `IF NOT EXISTS` skipou TODA a cláusula incluindo `REFERENCES`. FK ficou apontando para tabela `contratos` (inexistente — herança D-pre-8). INSERT falhava 100% silenciosamente.
**Lição:** Verificar `pg_constraint` pós-migration sempre que `ADD COLUMN IF NOT EXISTS` inclui `REFERENCES`. FK órfã é silenciosa em prod até feature consumir.
**Onde aplicar:** Toda phase com `ALTER TABLE ADD COLUMN IF NOT EXISTS ... REFERENCES ...`. Pattern Migration 005/006 idempotente: DROP CONSTRAINT IF EXISTS + ADD CONSTRAINT + verify + pg_notify.

### 6. Schema adapter long↔short form — replication em TODO consumer (D-pre-14)
**Contexto:** `calcularDetalheEncargosContrato` Phase 7.8 D-02 adapter lê encargos das dívidas individuais (`dividas`) — não do contrato (`contratos_dividas` declared single source). Drift sistêmico em 5 consumers.
**Lição:** Adapter DB↔motor com 2 fontes de mesma semântica (long form vs short form) deve ser reaplicado em TODO consumer com decisão explícita: qual fonte ganha? Pattern transferível: documentar single source em CONTEXT.md ANTES do adapter consolidar.
**Onde aplicar:** Phases com schema redundante (mesmo dado em 2+ tabelas). 7.13c BACKLOG fixará drift sistêmico do adapter encargos.

### 7. UAT humano pega bugs que gates automatizados não pegam (cumulative)
**Contexto:** Plan 02 PAUSA #1-RETOMADA descobriu 2 bugs em UAT humano:
- BUG #1 (FK schema): Migration 004 `ADD COLUMN IF NOT EXISTS` + `REFERENCES` silently skipped — INSERT falhava 100% silenciosamente.
- BUG #2 (saldo divergente): adapter long↔short form drift sistêmico afetando 5 consumers.
Plan 03 Task 3 fix descobriu 3º bug em UAT cumulative Tela 3:
- BUG #3 (useEffect histórico): rewrite UI Plan 02 não migrou `useEffect` `eventos_andamento` por `devedor_id` → `contrato_id`. Histórico vazio em UI (silencioso).
Gates automatizados Plan 01+02 (build + unit + grep + test:regressao) PASSARAM em todos esses casos.
**Lição:** UAT prod humano é detector primário pra adapter/FK/drift bugs + paridade comportamental UI. Gates não cobrem schema integrity (FK órfã), semantic drift (encargos), nem UI behavior parity (useEffect missing). Cadência transferível: a cada 3-4 phases de feature, UAT cumulative cross-tela obrigatório com console F12 limpo.
**Onde aplicar:** Phases que migram source of truth (devedor → contrato, etc) ou rewrite UI total — UAT cross-tela com console limpo é gate adicional além de gates automatizados.

### 8. State snapshot deve listar TODOS os commits da phase (Plan 02 → Plan 03 handoff)
**Contexto:** Plan 03 PAUSA pré-execute: snapshot original do orchestrator listava apenas Plan 02 atomic commit (325b964) sem mencionar Plan 01 (9b9e7ca). Decision tree assumia HEAD = Plan 02 sem confirmar Plan 01 ancestor.
**Lição:** State snapshots em PAUSAs entre plans devem listar TODOS os commits da phase em ordem topológica (Plan 01 → Plan 02 → Plan 03), não apenas o HEAD. Permite verificar parent chain íntegra.
**Onde aplicar:** Toda PAUSA inter-plan numa phase com 2+ plans atomic commits. State snapshot no formato `<plan_id>: <SHA> <subject>` por linha.

### 9. Rewrite UI deve auditar TODOS useEffects do HEAD (Plan 03 fix)
**Contexto:** Plan 03 Task 3 fix descobriu que rewrite UI Plan 02 não migrou `useEffect` `eventos_andamento` por `devedor_id` → `contrato_id`. Feature preservada em runtime (state inicializado com defaultValue), mas histórico ficava vazio em UI. Bug silencioso passou pelos gates Plan 02.
**Lição:** Rewrites UI totais devem auditar TODOS `useEffect`/`useMemo`/`useCallback` do HEAD — não apenas funções e components. Hooks com side-effects (fetch + setState) são fonte comum de drift silencioso.
**Onde aplicar:** Rewrite UI total — Plan PAUSA pre-merge inclui audit grep `useEffect.*\\[.*\\]` de HEAD vs rewrite, listando hooks ausentes/modificados.

### 10. FK audit pós-migration deve ser amplo, não pontual (Migration 006)
**Contexto:** Migration 005 fixou FK `eventos_andamento_contrato_id_fkey`. Migration 006 (2026-04-28) descobriu 2ª FK órfã (`fila_cobranca_contrato_id_fkey` apontando pra tabela `contratos` inexistente — mesma classe de bug D-pre-13). Audit pós-Migration 005 não foi wide o suficiente.
**Lição:** Quando uma FK órfã é descoberta, audit IMEDIATO de TODAS as FKs do schema (`SELECT * FROM pg_constraint WHERE confrelid::regclass::text NOT IN (SELECT tablename FROM pg_tables)` ou similar). 1 FK órfã sugere classe de bug, não incidente isolado.
**Onde aplicar:** Pós-fix de qualquer FK órfã ou após `ADD COLUMN IF NOT EXISTS ... REFERENCES`. Migration 006 fechou mapa de 7 FKs verificadas.

## Patterns (5 patterns reutilizáveis)

### Pattern 1: Migration γ DROP histórico legado robusta a 2 strategies (D-pre-7 + D-pre-10)
**Estrutura:** 5 blocos SQL — (1) ALTER ADD COLUMN nullable; (2) ALTER ADD COLUMN FK; (2.5) PRE-FLIGHT SELECT COUNT; (3) STRATEGY A `TRUNCATE` preferida + STRATEGY B `DELETE` fallback documentado; (4) ALTER SET NOT NULL pós-truncate; (5) `pg_notify('pgrst', 'reload schema')`. Decisão de strategy registrada no SUMMARY.
**Aplicado em:** Migration 004. Reutilizável para futuras γ-decisions onde dado histórico é descartável e banco é minúsculo.

### Pattern 2: Migration FK fix idempotente (Migration 005 + 006)
**Estrutura:** 5 blocos — (1) DROP CONSTRAINT IF EXISTS + integrity check (`SELECT count(*) FROM tabela WHERE FK_col IS NOT NULL`); (2) verifica tabela vazia ou rows válidas; (3) ADD CONSTRAINT REFERENCES tabela_certa(id) ON DELETE CASCADE; (4) verify (`SELECT * FROM pg_constraint WHERE conname=...`); (5) `pg_notify('pgrst', 'reload schema')`. Idempotente — pode ser re-rodada sem efeito colateral.
**Aplicado em:** Migration 005 (`eventos_andamento_contrato_id_fkey`), Migration 006 (`fila_cobranca_contrato_id_fkey`). Reutilizável para futuras FK órfãs descobertas em audit pg_constraint.

### Pattern 3: STATUS_CONTRATO export análogo a STATUS_DEV preservando D-02 isolation
**Estrutura:** 3 const arrays exportados em `constants.js` (`STATUS_CONTRATO`, `STATUS_CONTRATO_TERMINAIS`, `STATUS_CONTRATO_ATIVOS`). Consumido por service (filter ATIVOS no `entrarNaFila`) + UI (select com 7 options + toggle). D-02 isolation: status nunca lookuped via DB query, sempre via constant import.
**Aplicado em:** Plan 01 + Plan 02. Reutilizável para qualquer status enum com classificação (terminais/ativos/etc).

### Pattern 4: D-02 adapter contrato-level reusável (D-pre-12)
**Estrutura:** Wrapper thin `calcularSaldoContratoAtualizado(d, p, h) = calcularDetalheEncargosContrato(d, p, h).saldoAtualizado` em `devedorCalc.js` ao lado do adapter Phase 7.8 (D-02). Motor base D-01 INTOCADO.
**Aplicado em:** Plan 01. Reutilizável para qualquer feature contrato-level que precisa de campo extraído do shape do adapter — não duplica adapter, compõe.

### Pattern 5: Rewrite total absorve fix-forward sem isolar como sub-phase (D-pre-8)
**Estrutura:** Bugs latentes em features dormentes (`filaDevedor.js` apontando pra `contratos` inexistente, dormente em prod por meses) são fixados pelo rewrite total estrutural — sem isolar como sub-phase 7.13c. Documentar como deviation no SUMMARY com lição transferível.
**Aplicado em:** Plan 01 (5+ stale `dbGet("contratos", ...)` callsites). Reutilizável para refactors estruturais onde audit pré-execute revela features dormentes com bugs silenciosos.

## Surprises (5 surpresas durante execução)

**Lição transferível:** Adapter long↔short form precisa decisão explícita de qual fonte ganha — drift silencioso até UAT cross-feature.

### Surprise 3: Botão "Atender" handler signature mismatch (Plan 02 Opção A)

**Lição transferível:** Rewrite UI consume service refactored exige audit de signature handler-by-handler. "Service legacy faz a coisa certa" é assumption frágil.

### Surprise 4: useEffect histórico eventos missing em rewrite UI (Plan 03 Task 3)

**Resolução:** Plan 03 fix em `FilaDevedor.jsx` L466-474 — useEffect com filter `contrato_id`. Schema 7.13b já garante `contrato_id` populated.
**Lição transferível:** Rewrite UI total deve auditar TODOS `useEffect` do HEAD — não apenas funções/components. Hooks com side-effects são drift silencioso.

### Surprise 5: 2ª FK órfã descoberta pós-Migration 005 (Migration 006)
**Discovery:** UAT cumulative 2026-04-28 Tela 2 (FilaOperador) — feature relacionada a `fila_cobranca` falhou. Audit revelou `fila_cobranca_contrato_id_fkey` apontando para tabela `contratos` inexistente — mesma classe de bug D-pre-13 mas em outra FK.
