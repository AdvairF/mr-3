## Plan 03 Task 3 — useEffect histórico eventos (D-pre-15)

**Bug detectado em UAT cumulative Tela 3:** ao abrir contrato em FilaAtendimento, listagem de histórico de eventos ficava vazia. Investigação: rewrite Plan 02 não migrou `useEffect` que existia no HEAD pre-7.13b (`FilaDevedor.jsx` L570-581) que fetchava `eventos_andamento` filtrado por `devedor_id`. Feature preservada em runtime (state inicializado com defaultValue), mas histórico não populava da DB.

**Fix aplicado:** `useEffect` em `FilaDevedor.jsx` L466-474:
```js
// Carrega histórico de eventos do banco no mount + on contrato.id change
// Paridade comportamental com HEAD pre-7.13b L570-581 (filter migrado para contrato_id — schema 7.13b D-pre-10).
// Limit=50 preservado de HEAD por paridade — paginação/limit configurable é phase futura se demandado.
useEffect(() => {
  if (!contrato?.id) return;
  dbGet("eventos_andamento", `contrato_id=eq.${contrato.id}&order=data_evento.desc&limit=50`)
    .then(rows => { if (Array.isArray(rows)) setEventos(rows); })
    .catch(err => {
      console.error("Erro ao fetchar eventos_andamento do contrato:", contrato?.id, err);
    });
}, [contrato?.id]);
```

Schema 7.13b D-pre-10 já garante `contrato_id` populated em rows novas — filter funcional. Limit=50 preservado de HEAD por paridade. Bug silencioso passou pelos gates Plan 02 (build + unit + grep + test:regressao) — só detectado em UAT cumulative humano.

Promovido a **D-pre-15 POST-HOC** no CONTEXT.md L125 via Plan 03 Etapa 5 (governança: decisão arquitetural com fix aplicado em prod merece âncora primária no CONTEXT.md, não viver apenas no LEARNINGS — memory feedback `feedback_q_marks_must_be_promoted_to_dpre.md` registra padrão).

## Migration 006 — FK fix `fila_cobranca_contrato_id_fkey` (D-pre-13 cumulative)

**Fix aplicado em prod 2026-04-28** manualmente via Supabase SQL Editor (5 blocos sequenciais):

| Bloco | Ação | Idempotente |
| ----- | ---- | ----------- |
| 1     | `DROP CONSTRAINT IF EXISTS fila_cobranca_contrato_id_fkey` | ✅ |

| 3     | `ADD CONSTRAINT ... REFERENCES contratos_dividas(id) ON DELETE CASCADE` em `DO $$` block com check pg_constraint | ✅ |
| 4     | Verify (`SELECT pg_get_constraintdef`) confirmando FK target correto | ✅ (read-only) |
| 5     | `pg_notify('pgrst', 'reload schema')` | ✅ |

Tabela vazia em prod 2026-04-28 confirmada via integrity check (Bloco 2) — ADD CONSTRAINT seguro.
