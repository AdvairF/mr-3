- **100% incidência** no fluxo Fila / Carteira Total / DetalheContrato — todos os 5 consumers do adapter herdam drift
- **0 rows de heterogeneidade intra-contrato** — encargos são uniformes dentro de cada contrato; só divergem do header

Detalhes completos: `07.13c-AUDIT-PROD.md`.

## 2. Decisão arquitetural locked: Path A puro

**Path A — Motor lê do contrato via adapter (refactor adapter)**

### Mecânica

- `calcularDetalheEncargosContrato(contrato, dividasDoContrato, allPagamentosDivida, hoje)` — assinatura nova com `contrato` como 1º argumento
- Adapter sobrescreve campos de encargo no `pseudoDevedor.dividas` ANTES de delegar ao motor
- Motor `calcularDetalheEncargos` continua **INTOCADO** (D-01 cumulative cumprido — devedorCalc.js:365-521 zero diff)

### Pseudo-código

```js
function calcularDetalheEncargosContrato(contrato, dividasDoContrato, allPagamentosDivida, hoje) {
  const dividaIds = new Set((dividasDoContrato || []).map(d => String(d.id)));
  const pagamentosFiltrados = (allPagamentosDivida || []).filter(p => dividaIds.has(String(p.divida_id)));

  const dividasComEncargosDoContrato = (dividasDoContrato || []).map(d => ({
    ...d,
    indexador:      contrato.indice_correcao,
    juros_tipo:     contrato.juros_tipo,
    juros_am:       contrato.juros_am_percentual,
    multa_pct:      contrato.multa_percentual,
    honorarios_pct: contrato.honorarios_percentual,
    art523_opcao:   contrato.art523_opcao,
    // data_inicio_atualizacao + custas + valor_total + data_vencimento — NÃO sobrescrever (per-DIVIDA)
  }));

  const pseudoDevedor = { dividas: dividasComEncargosDoContrato };
  return calcularDetalheEncargos(pseudoDevedor, pagamentosFiltrados, hoje);
}
```

### Justificativas para Path A vs B vs C

| Critério | Path A | Path B (cascade) | Path C (override flag) |
|---|---|---|---|
| Heterogeneidade intra-contrato (Q1 audit) | Suportada (zero ocorrência hoje) | Quebra | Suporta com complexidade |
| Migration de dados retroativa | Não exige | Obrigatória (risco financeiro) | Migration nova de schema |
| EDT-04 (Phase 6) reconciliação | Cooperativa (D-pre-6) | Revogação direta | Cooperativa |
| Tamanho do escopo | 2-3 plans | 3-4 plans | 4+ plans |
| Risco de drift novo | Eliminado (motor lê do contrato) | Reaparece se cascade falha mid-flight | Eliminado por flag |

Audit confirmou Q1=0 rows + Q2=fonte única declarada → **Path A puro é o caminho mais limpo**.

## 3. Decisões pre-locked (D-pre-1 a D-pre-7)

### D-pre-1 — 7 campos de encargo per-CONTRATO

Contrato é fonte única para os seguintes 7 campos de encargo:

| Campo (long form / contratos_dividas) | Campo (short form / motor) |
|---|---|
| `indice_correcao` | `indexador` |
| `juros_tipo` | `juros_tipo` |
| `juros_am_percentual` | `juros_am` |
| `multa_percentual` | `multa_pct` |
| `honorarios_percentual` | `honorarios_pct` |
| `art523_opcao` | `art523_opcao` |
| `despesas` | (motor não consome — preservado para reuso futuro) |

Adapter sobrescreve estes campos no `pseudoDevedor.dividas` a partir do `contrato`.

### D-pre-2 — 4 campos permanecem per-DIVIDA

Os seguintes campos **NÃO** são sobrescritos pelo adapter — permanecem propriedade legítima da divida individual:

| Campo | Justificativa |
|---|---|
| `data_inicio_atualizacao` | Cada parcela tem início próprio; motor já fallbackeia `\|\| div.data_vencimento \|\| div.data_origem` (devedorCalc.js:387) |
| `custas` (JSONB array) | Custas judiciais são lançamentos por divida, não por contrato |
| `valor_total` | Parcelamento — cada divida tem valor próprio |
| `data_vencimento` / `data_origem` | Por construção, cada parcela tem vencimento próprio (gerado por gerarPayloadParcelasDocumento) |

### D-pre-3 — Standalone, enriquecimento e callsites com contrato explícito

#### D-pre-3.1 — Standalone (`contrato_id IS NULL`)

Dividas com `contrato_id IS NULL` (criadas via `App.jsx:3247-3264` standalone, fluxos forenses 7.10 — custas avulsas, dividas isoladas) usam **encargos próprios da divida** (back-compat fluxo forense 7.10). Adapter e populator pulam sobrescrita quando `contrato_id` é NULL.

#### D-pre-3.2 — Enriquecimento no populator (`dividasMap` em App.jsx:8418-8428)

Dividas com `contrato_id` não-NULL consumidas via `App.jsx:2040` (Pessoas/Detalhe-Devedor → `calcularDetalheEncargos` direto, **sem adapter**) são enriquecidas no `dividasMap` populator:

- Lookup em `allContratos` por `div.contrato_id`
- Sobrescreve os 7 campos de encargo (D-pre-1) na divida enriquecida
- Strategy A wrapper delegation pattern (memory feedbacks `schema_adapter_long_short_form_consumer_replication` + helper-first)

Beneficiário único: `App.jsx:2040` (Pessoas/Detalhe-Devedor). Demais consumers usam o adapter (D-pre-3.3).

#### D-pre-3.3 — Callsites de adapter recebem contrato explícito (1º arg)

Os 5 callsites de `calcularDetalheEncargosContrato` recebem `contrato` como **1º argumento explícito** — não duplicam lookup do populator:

- `useSaldoAtualizadoCache.js:64`
- `filaDevedor.js:148, 241`
- `FilaDevedor.jsx:480`
- `DetalheContrato.jsx:589`
- (e o 5º via Pessoas atendido pelo populator D-pre-3.2 — não precisa argumento adicional)

Adapter faz override sobre as dividas recebidas, sem depender do enriquecimento prévio.

**Justificativa para arquitetura dupla (populator + adapter explícito):**
- Populator centraliza enriquecimento para o caminho de motor direto (`App.jsx:2040`)
- Adapter recebe contrato para os 4 callsites onde contrato já está no scope (DetalheContrato/Fila/Cache)
- Sem duplicação: cada divida é enriquecida UMA vez no caminho correspondente
- Caminhos não se cruzam: populator cuida do path Pessoas; adapter cuida dos 4 paths Contrato/Fila

### D-pre-6 — EDT-04 (Phase 6) reconciliada

EDT-04 dizia "encargos funciona como template — não retroage em parcelas já geradas" (`contratos.js:140`).

**Reconciliação pós-7.13c:**

- **Leitura de saldo** passa a ser do contrato (fonte única) — adapter sobrescreve no read
- **Edição de contrato** continua sem cascade (`editarContrato` em `contratos.js:142-144` zero diff) — não precisa cascade pois leitura agora pula direto pro contrato

EDT-04 não é revogada — é refinada: "encargos é template **na escrita**, mas leitura é **do contrato**".

### D-pre-7 — Comparison report pré-deploy (SQL puro + formato + abort gate)

#### D-pre-7.1 — Implementação em SQL puro

Comparison report Plan 02 implementado em **SQL puro no Supabase**, **NÃO** em script Node ad-hoc.

**Justificativa:**
- SQL reproduz lógica de encargos diretamente no banco (verificável, sem risco de divergência runtime JS)
- Auditável por humano (advogado lê SQL melhor que JS)
- Sem setup de runtime separado
- Trade-off: SQL fica comprido. Aceito por auditabilidade.

#### D-pre-7.2 — Formato do report

Para cada um dos 3 contratos com drift, o comparison report deve mostrar:

- **Saldo principal antes (drift)** — `dividas.*_percentual` aplicado sobre `dividas.valor_total`
- **Saldo principal depois (Path A)** — `contratos_dividas.*_percentual` aplicado sobre `dividas.valor_total`
- **Delta absoluto** (R$) e **percentual** (% do saldo antes)
- **Mesma estrutura separada** para: juros, multa, honorários, correção monetária

Formato exemplo:

```
Contrato M L FRIOS (335a2ad2):
  Principal:  Antes R$ X     | Depois R$ Y     | Δ R$ Z (P%)
  Juros:      Antes R$ Xj    | Depois R$ Yj    | Δ R$ Zj (Pj%)
  Multa:      Antes R$ Xm    | Depois R$ Ym    | Δ R$ Zm (Pm%)
  Honor.:     Antes R$ Xh    | Depois R$ Yh    | Δ R$ Zh (Ph%)
  Correção:   Antes R$ Xc    | Depois R$ Yc    | Δ R$ Zc (Pc%)
  TOTAL:      Antes R$ T1    | Depois R$ T2    | Δ R$ ΔT (PT%)
```

Output em `07.13c-COMPARISON.md` (Plan 02 deliverable). Smoke prod do Plan 03 valida que UI exibe os novos valores correspondentes ao "Depois".

#### D-pre-7.3 — ABORT GATE

Se delta surpreender o orchestrator em qualquer dos 3 contratos (exemplo: M L FRIOS pula de R$ 55k para R$ 200k — magnitude desproporcional ao drift de encargos):

- Deploy **ABORTA** antes do tag `v1.4-phase7.13c`
- Phase volta a `/gsd-discuss-phase` para revisar D-pre-1 (campos errados na partição CONTRATO) ou D-pre-2 (campos errados na partição DIVIDA)
- Plan 02 **NÃO** é descartado — comparison report já gerado serve de input pra discuss revisado
- Deviation registrada em SUMMARY do Plan 02 com hipótese de causa (campo mal-classificado, fórmula divergente, edge case prod, etc.)
