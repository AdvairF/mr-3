## 1. Problema

Phase 7.2 deu ao usuário um botão "Excluir Contrato" para apagar contrato inteiro. Mas não há equivalente para apagar UM documento dentro do contrato. Se o advogado cadastra um documento errado (tipo, valor, parcelas), a única saída hoje é apagar e recriar o contrato inteiro — drástico demais pra erro de digitação num doc.

## 2. Escopo aprovado

**Fix A** — Service layer (`src/services/contratos.js`):
- Nova função `excluirDocumento(documentoId)` — **espelha 100% o padrão de `excluirContrato`** (contratos.js:329-371)
- Pre-check 1: nenhum `pagamentos_divida` nas parcelas do documento (proteção contra apagar silenciosamente registros de pagamento)
- Pre-check 2: **NÃO** checa `pagamentos_contrato` (o nível contrato permanece intocado; SP-generated amortizations já estão em `pagamentos_divida` → captadas pelo pre-check 1)
- Cascata manual: `DELETE dividas WHERE documento_id=:id` → `DELETE documentos_contrato WHERE id=:id` (FK CASCADE cuida de `pagamentos_divida` + `devedores_dividas`)
- **Pós-delete**: chama `recalcularTotaisContrato(contratoId)` — senão `contrato.valor_total` + `num_documentos` + `num_parcelas_total` ficam stale (mesmo helper que `adicionarDocumento:283` já usa)
- Retorna `{ ok: true }` ou `{ ok: false, motivo: string }`

**Fix B** — UI (`src/components/DetalheContrato.jsx`):
- Botão "Excluir documento" vermelho outline, pequeno, no mesmo `<div>` que contém "Editar parcelas" (linha ~765)
- `handleExcluirDocumento(doc)`:
  - `window.confirm("Tem certeza? Esta ação não pode ser desfeita.")` — mesma string da Phase 7.2
  - **UI state cleanup ANTES do delete**: se `editingDocId === doc.id` → `setEditingDocId(null)`; se `expandedDoc === doc.id` → `setExpandedDoc(null)`. Previne render crash pós-reload quando card expandido aponta pra doc inexistente.
  - Chama `excluirDocumento(doc.id)` → toast success/error
  - `await onCarregarTudo()` + refresh local de `documentos` (padrão `handleDocumentoAdicionado:404-408`)
  - **NÃO** navega `onVoltar()` (diferente da 7.2 que sai do DetalheContrato — aqui ficamos no contrato, só o documento some)

## 4. Decisões locked (discuss 2026-04-23)

- **D-01** — Motor Art.354 (`utils/devedorCalc.js`, `services/pagamentos.js`) **NÃO** tocado. 9/9 regressão TJGO preservada. Phase é pure CRUD — nenhum teste novo adicionado.

- **D-02** — Template da Phase 7.2 (`excluirContrato` + `handleExcluirContrato`) copiado 1:1 em termos de estrutura: pre-check → manual cascade → return. Apenas **filtro** muda (`documento_id` em vez de `contrato_id`) e **ordem final** adiciona `recalcularTotaisContrato`.

- **D-03** — Pre-check é **safety feature**, não requirement técnico. FK `pagamentos_divida.divida_id REFERENCES dividas(id) ON DELETE CASCADE` faria a cascata silenciosa; o pre-check protege contra apagar registros de pagamento sem intenção explícita.

- **D-04** — `DELETE documentos_contrato` **FALHA com FK violation** se existirem `dividas` com `documento_id` apontando pra ele (NO CASCADE em `dividas.documento_id`). Portanto: `DELETE dividas` SEMPRE antes de `DELETE documentos_contrato`. Ordem é invariante.

- **D-05** — `recalcularTotaisContrato(contratoId)` **OBRIGATÓRIO** pós-delete. Sem isso, `contrato.valor_total`, `num_documentos`, `num_parcelas_total` ficam stale. Chamar DEPOIS dos DELETEs e ANTES do return `{ ok: true }`.

- **D-06** — `pagamentos_contrato` **NÃO** entra no pre-check. Amortizações via SP já geram rows em `pagamentos_divida` (SP atômica). Pre-check em `pagamentos_divida` captura tanto pagamentos manuais quanto SP-generated. Checar `pagamentos_contrato` separadamente seria redundante.

- **D-07** — **UI state cleanup** ANTES do delete: `if (editingDocId === doc.id) setEditingDocId(null);` + `if (expandedDoc === doc.id) setExpandedDoc(null);` — previne render crash quando `onCarregarTudo` retorna lista sem o doc.

- **D-08** — Contrato vazio (num_documentos=0) pós-delete do último doc é **estado válido**. Simétrico com fluxo "criar contrato, adicionar doc depois" que já permite contrato sem docs. Sem tratamento especial.

- **D-09** — `contratos_historico` NÃO bloqueia exclusão (referencia `contrato_id`, não `documento_id`). Evento forense de exclusão DEFERIDO pra phase futura de rastreabilidade.

## 7. Achados da investigação (pré-planning)

### Schema FK analysis

| Tabela.coluna | Target | ON DELETE |
|---|---|---|
| `documentos_contrato.contrato_id` | `contratos_dividas(id)` | **NO CASCADE** |
| `dividas.documento_id` | `documentos_contrato(id)` | **NO CASCADE + NULLABLE** |
| `pagamentos_divida.divida_id` | `dividas(id)` | **CASCADE ✓** |
| `devedores_dividas.divida_id` | `dividas(id)` | **CASCADE ✓** |

Consequência: `dividas` tem que ir antes de `documentos_contrato`. `pagamentos_divida` + `devedores_dividas` caem automaticamente quando `dividas` morre.

### Template Phase 7.2

`excluirContrato` em `contratos.js:329-371`: 2 pre-checks + 3 DELETEs + return. Nossa função mira estrutura idêntica, só troca filtros e adiciona `recalcularTotaisContrato` no final.

`handleExcluirContrato` em `DetalheContrato.jsx:385-402`: window.confirm → service call → toast + `onCarregarTudo` + `onVoltar`. Nossa handler **não** chama `onVoltar` (fica no contrato).

### Botão "Editar parcelas"

Localização: `DetalheContrato.jsx:766-774` dentro do card expandido, em `<div style={{ marginBottom: 8, textAlign: "right" }}>`. Nosso botão "Excluir documento" vai NO MESMO div, à esquerda ou à direita (decisão cosmética do planner). Padrão Phase 7.2 usa vermelho outline small Btn.

### Helper de recalculo

`recalcularTotaisContrato(contratoId)` em `contratos.js:219-226`. Já usado por `adicionarDocumento:283`. Nossa função chama no mesmo lugar (final do fluxo, antes do return `{ ok: true }`).
