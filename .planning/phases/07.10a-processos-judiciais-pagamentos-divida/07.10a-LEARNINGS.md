# Phase 7.10a Learnings: processos-judiciais-pagamentos-divida

## Decisions

### D-26 ProcessosJudiciais lê apenas pagamentos_divida (Shield 22)
Callsite App.jsx L8478 fixa `pagamentos={allPagamentosDivida}` no `case "processos"`. Shield 22 (positive grep + regional negative awk 10 linhas) blinda contra refactor futuro acidentalmente trazer `allPagamentos` legacy de volta.

**Rationale:** Single-source-of-truth pra ProcessosJudiciais — pagamentos via stored proc Phase 7 (em `pagamentos_divida`) é a fonte correta. Shield grep estrutural blinda regressão silenciosa pós-merge.
**Source:** 07.10a-CONTEXT.md §4 (D-26), 07.10a-01-PLAN.md Tasks 1+4

### D-27 Filter via lookup divida_id ∈ dividas-do-devedor (Set lookup pattern)
ProcessosJudiciais.jsx L254-256: `const dividaIdsDoDevedor = new Set((fullDev?.dividas || []).map(d => String(d.id)))`; `pagamentos.filter(p => dividaIdsDoDevedor.has(String(p.divida_id)))`. NÃO usa `devedor_id` direto.

**Rationale:** `pagamentos_divida` schema (de pagamentos.js L7-14 comment) tem apenas `divida_id` como FK. `devedor_id` direto não existe no shape novo. Set lookup é O(1) per pagamento, total O(n) onde n = total pagamentos do contrato.
**Source:** 07.10a-CONTEXT.md §4 (D-27), 07.10a-01-PLAN.md Task 2

### D-28 Drift histórico aceito por design
Pagamentos via UI legacy "Registrar pagamento parcial" (App.jsx L2537/2558/2574) ainda escrevem em `pagamentos_parciais`. Após 7.10a, ProcessosJudiciais lê `pagamentos_divida` — pagamentos novos via UI legacy NÃO aparecem em ProcessosJudiciais até 7.10b/c/d migrarem o resto OU phase posterior unificar writes.

**Rationale:** Cortar writes é phase própria (7.10e hipotético). Misturar com migração de leitura quebra atomicidade — escopo 7.10a fica pequeno e auditável. Trade-off: drift visível por design, documentado em SUMMARY visivelmente pro usuário.
**Source:** 07.10a-CONTEXT.md §4 (D-28), 07.10a-01-SUMMARY.md seção "UAT cross-check empírico"

### D-29 Motor compatível verificado (data_pagamento + valor only)
`calcularSaldoDevedorAtualizado` (devedorCalc.js L59) lê apenas `p.data_pagamento + p.valor`. Não consulta `devedor_id` nem `divida_id`. Shape `pagamentos_divida` (que tem ambos data_pagamento+valor) é compatível. Motor distribui pagamentos FIFO pelas dividas do devedor — mesmo comportamento de antes.

**Rationale:** D-01 strict garantido por construção — motor não precisa mudar. Compatibilidade verificada via leitura direta do source pré-implementação (Q3 da discuss).
**Source:** 07.10a-CONTEXT.md §4 (D-29), discuss-phase Q3

Dashboard (10 refs em agregados visíveis) deferido pra 7.10d como mais arriscado.

## Lessons

Phase 7.10a SC2 originalmente exigia "Screenshot prod saldo (cross-check pré-migração)". Bug pré-existente em prod (`[object Object]` typo no nome de tabela em dbInsert "processos_judiciais") impediu cadastro de processo de teste — UAT cross-check empírico real impossível.

**Context:** Lição registrada em `memory/feedback_proc_judiciais_create_bug_preexisting.md`. Em discuss/plan-checker pra phases de migração de consumidor, adicionar checklist obrigatório: (1) feature alvo utilizável em prod hoje? (abrir prod, executar fluxo principal). (2) Telemetria de uso real (Total=0?). (3) Bug pré-existente descoberto durante UAT é Phase X.bug standalone, não fail da phase atual.
**Source:** 07.10a-01-SUMMARY.md "UAT cross-check empírico" section, feedback_proc_judiciais_create_bug_preexisting.md

**Context:** Trade-off aceito: ship phase com risco residual conhecido (sem cross-check empírico real), monitorar smoke prod com cuidado redobrado. Shield 23 cobre semantic correctness sem dados de prod. Pattern reutilizável pra phases futuras onde feature alvo está broken em prod ou tem Total=0.

**Context:** Phases 7.10b/c/d (Devedores, ModuloDividas, Dashboard) têm features alvo utilizáveis em prod (com dados reais) — pre-check trivial pra essas, UAT cross-check empírico real será viável. ProcessosJudiciais foi outlier.

## Patterns

### Set lookup filter pra `divida_id ∈ dividas-do-devedor`
```js
const dividaIdsDoDevedor = new Set((fullDev?.dividas || []).map(d => String(d.id)));
const pgtosDev = pagamentos.filter(p => dividaIdsDoDevedor.has(String(p.divida_id)));
```

**When to use:** Consumer precisa filter pagamentos_divida por devedor (sem `devedor_id` direto na nova shape). Pattern reutilizável pra 7.10b (Devedores), 7.10c (ModuloDividas), 7.10d (Dashboard) se replicarem necessidade. Se 2+ consumers replicarem, extrair pra helper compartilhado `src/utils/filterPagamentosDoDevedor.js`.
**Source:** 07.10a-CONTEXT.md §2 Fix B, ProcessosJudiciais.jsx L254-256

### Shield 23 equivalência semântica fixture sintética
Test compara pattern novo vs pattern legacy sintético (injeta o campo legacy no shape novo, valida que filters retornam mesmos itens). Cobre semantic correctness sem depender de dados reais de prod.

**When to use:** Phases que mudam shape API (campo X removido/renomeado/movido) ou pattern de filter. Cobre H3-class shape mismatch silencioso. Custo baixo (~15 linhas test). Lição central da Phase 7.8.2a (commit 504e83e).
**Source:** processosJudiciaisFilter.test.js Shield 23, feedback_shields_wrapper_must_test_equivalence.md

### Memory feedback indexed por phase
Cada lição que generaliza além da phase atual vira arquivo `memory/feedback_<topic>.md` com YAML frontmatter (name, description, type) + linha em `MEMORY.md` (índice de uma linha). 7.10a adiciona 1 entry: `feedback_proc_judiciais_create_bug_preexisting.md`. Total: 14 entries indexadas.

**When to use:** Lição que se aplica a próximas phases (regra/protocolo). Pattern continua de 7.9 (que adicionou 3 entries).
**Source:** memory/MEMORY.md L17, memory/feedback_proc_judiciais_create_bug_preexisting.md

### PAUSAs blocking proporcionais ao risco
Phase 7.9 (escopo grande, 7 files commit base + 6 fix-forwards): 3 PAUSAs (UAT + diff + bump review). Phase 7.10a (escopo pequeno, 4 files single commit + Shield 23 equivalência): 2 PAUSAs (UAT + diff; SHA cross-check incorporado em Task 3 `<verify>`).

**When to use:** Phases pequenas + cobertura automatizada equivalente ao gate humano podem omitir PAUSA #3 (bump review). Documentar racional no plan body. Phases com risco alto + complexidade alta retêm 3 PAUSAs.
**Source:** 07.10a-02-PLAN.md objective + Task 2 what-built justificativa

### UAT empírico simplificado quando cross-check real é impossível
Quando feature alvo está broken em prod ou tem Total=0 (sem dados pra cross-check), pivot pra: (1) feature renderiza no local + console limpo, (2) confiança em shields automatizados (estrutural + semântica). Cross-check valor SKIPPED com nota explícita em SUMMARY.

**When to use:** Phases de migração de consumidor onde feature alvo não é utilizável em prod. Trade-off documentado pro usuário (drift residual aceito).
**Source:** 07.10a-01-SUMMARY.md "UAT cross-check empírico" section

### Pre-existing bug detection during UAT registers as Phase X.bug
Bug descoberto durante UAT da phase atual mas independente do escopo dela → registrar Phase X.bug standalone em ROADMAP backlog (sem prioridade fixa). Não é fail da phase atual; phase atual pivota pra UAT alternativo + ship.

**When to use:** Bug é independente do escopo da phase (não é regressão da migração) e bloqueia UAT empírico real. Documentar em memory feedback + backlog entry com `Depends on`/`Blocks` indicando relação com phases futuras.
**Source:** ROADMAP.md Phase 7.10.bug entry, feedback_proc_judiciais_create_bug_preexisting.md

## Surprises

### ProcessosJudiciais cadastro nunca funcionou em prod (typo `[object Object]`)
Bug pré-existente descoberto durante UAT 7.10a (PAUSA #1): tela "+ Novo Processo" gera erro `Could not find the table 'public.processos_judiciais[object Object]' in the schema cache`. Typo JS clássico (concat objeto com string em algum `dbInsert`). Tela ProcessosJudiciais Total=0 em prod confirma feature nunca funcionou.

**Impact:** UAT cross-check empírico real impossível com a estratégia original. Pivot pra UAT empírico simplificado + shields automatizados. Bug registrado como Phase 7.10.bug em ROADMAP backlog. Lição em `feedback_proc_judiciais_create_bug_preexisting.md`.
**Source:** 07.10a-01-SUMMARY.md "UAT cross-check empírico" section, ROADMAP.md Phase 7.10.bug

### Total=0 em prod validou retroativamente escolha "mais simples" pra 7.10a
ProcessosJudiciais foi escolhido como 7.10a baseado em superfície reduzida + folha + sem testes acoplados. Descobriu-se em UAT que era também o único consumidor com feature alvo broken em prod. "Mais simples" e "ninguém usa" coincidiram fortuitamente — risco de regressão visível foi MUITO baixo.

**Impact:** Phases 7.10b/c/d (Devedores/ModuloDividas/Dashboard) têm features utilizáveis com dados reais em prod — UAT cross-check empírico real será viável. ProcessosJudiciais foi outlier defendendo a estratégia atômica de phases pequenas.
**Source:** 07.10a-01-SUMMARY.md, ROADMAP.md Phase 7.10a entry

### Schema `pagamentos_divida` documentado apenas em JSDoc comment (sem migration SQL formal)
Tabela criada em Phase 7.3 mas migration SQL ficou como comentário em `pagamentos.js` L7-14 (JSDoc), não em `src/services/migrations/00X_*.sql`. Migrations existentes (`001_devedores_dividas.sql`, `002_dividas_tabela.sql`) não criam pagamentos_divida — foi executado manualmente no Supabase SQL Editor sem ser comitado como artifact SQL.

**Impact:** Schema discovery exigiu leitura do JSDoc em pagamentos.js durante discuss-phase Q2. Pattern arqueológico do projeto — nem toda tabela tem migration SQL no repo. Phases futuras: assumir que CREATE TABLE pode estar documentado apenas em comments de service files.
**Source:** discuss-phase Q2 investigation, pagamentos.js L7-14

### Motor `calcularSaldoDevedorAtualizado` lê apenas data_pagamento + valor (zero acoplamento a campos extras)
Antes da implementação, suspeita: motor pode consultar `devedor_id` ou `divida_id` do array de pagamentos pra distribuição FIFO. Verificação em devedorCalc.js L59-: motor itera devedor.dividas filtradas e processa pagamentos in-order via `data_pagamento + valor`. Distribuição FIFO é por dívidas do devedor (ordem do array), não por chave dos pagamentos.

**Impact:** Shape `pagamentos_divida` (que tem ambos data_pagamento+valor) é compatível direto. Zero motor edit. D-01 strict garantido por construção, sem precisar relaxar.
**Source:** discuss-phase Q3 investigation, devedorCalc.js L59-

### Existem 2 tabelas `contratos` no projeto (já era surpresa da 7.9, reaparece como contexto)
`contratos` (legado, usado por filaDevedor.js) coexiste com `contratos_dividas` (v1.4 atual, usado por contratos.js). Já era surpresa documentada em 7.9-LEARNINGS.md. Em 7.10a, irrelevante diretamente (consumer não toca contratos), mas cuidado redobrado em phases que tocam services + queries diretas: confundir uma pela outra é silencioso (retorna [] sem erro de tabela inexistente).

**Impact:** Pattern arqueológico do projeto. Phases futuras 7.10b/c/d devem manter cuidado quando tocarem services. Documentado em `feedback_db_integration_gate_missing.md` lessons da 7.9.
**Source:** 7.9-LEARNINGS.md (carry-over), feedback_db_integration_gate_missing.md
