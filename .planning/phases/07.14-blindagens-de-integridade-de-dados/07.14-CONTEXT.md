# Phase 7.14 — Blindagens de Integridade de Dados

**Milestone:** v1.4 — Pagamentos por Contrato + PDF Demonstrativo
**Modo:** standard reforçado, autonomous: false, múltiplos commits atômicos (pattern 7.10bcd / 7.13)
**Origem:** backlog 2026-04-26 — sub-item 1 = bug bloqueante DELETE devedor em prod (FK violation `eventos_andamento_devedor_id_fkey`); sub-itens 2/3/4 = blindagens de schema validation que surfaceram durante UAT 7.13 e phases anteriores
**Discuss data:** 2026-04-27

## Objetivo

Fechar lacunas de validação **pré-existentes** que afloraram quando features novas (7.13 múltiplos devedores) começaram a exercitar caminhos pouco testados:

1. DELETE físico de devedor falha em prod por FK não-CASCADE → soft delete via `deleted_at`
2. Botão "+ Nova Dívida" cria dívida fora do contrato (caminho legacy obsoleto pós-7.5/7.6/7.13)
3. Criação de contrato permite `devedor_id`/`credor_id` null silenciosamente
4. Inputs de data, valor monetário, CPF/CNPJ, CEP e telefone não aceitam paste de texto BR (`DD/MM/AAAA`, `R$ 1.234,56`, etc.) → fricção no fluxo do usuário

Lição central pré-execute (feedback `blindagens_integridade_apos_funcionamento`): phases que adicionam features expõem lacunas de validação que ficaram dormentes. Pre-execute walkthrough deve incluir "tentar quebrar" — mas como não fizemos isso em phases anteriores, o débito acumulado vai pra esta phase de blindagem.

## 4 Sub-itens (semânticamente conectados, 1 phase única)

| # | Sub-item | Tamanho | Risco |
|---|----------|---------|-------|
| 1 | Soft delete devedor (`deleted_at TIMESTAMPTZ NULL`) | Médio (1 migration + 1 helper UPDATE + 4 callsites filtro + UI confirmação) | Baixo (banco minúsculo: 6 devedores) |
| 2 | Remover botão "+ Nova Dívida" do ModuloDividas | Trivial (1 botão deletado, 1 view eliminada) | Baixo (zero órfãos: `dividas.contrato_id IS NULL = 0`) |

| 4 | Paste-friendly BR (`parseInputBR.js` + wrapper `<InputBR>`) | Médio (helper novo + wrapper + integração em ~24 inputs date+value; CPF/CNPJ/CEP/tel mantém mask existente) | Médio (varredura ampla, risco de regressão em encargos `parseFloat`) |

**Estimativa:** 1-2 sessões médias. Mais simples que 7.13 (zero migrations destrutivas, zero motor mexido, zero data backfill).

## Decisões locked (D-pre)

### D-pre-1 — Soft delete escopo MÍNIMO: devedores apenas

Coluna `deleted_at TIMESTAMPTZ NULL` em `devedores`. Credores/contratos/processos NÃO entram nesta phase.

**Why:** usuário só reportou bug bloqueante em pessoas/devedores. Credores hard delete continua funcionando hoje (sem FK violations conhecidas). Phase futura se aparecer demanda real.
**How to apply:** todas alterações de schema + filtros aplicam-se SOMENTE a `devedores`. Não tocar outras tabelas.

### D-pre-2 — Validação contrato: SOMENTE criação NOVA, legacy preservado

Criar contrato sem devedor/credor: bloqueado. Editar contrato legacy órfão: permitido (não há órfãos em prod hoje, mas regra fica defensiva).

**Why:** usuário não quer migration agressiva NOT NULL no schema (zero órfãos confirmados, mas evita risco). Validação app-level é reversível, schema NOT NULL não é.
**How to apply:** schema mantém `contratos_dividas.devedor_id` e `contratos_dividas.credor_id` nullable. Validação ocorre em (a) UI `NovoContrato.handleSalvar` e (b) service-layer `criarContrato` em `services/contratos.js`.

### D-pre-3 (REVISTA) — Paste-friendly BR escopo: APENAS date + value via `<InputBR>`

CPF/CNPJ/CEP/telefone permanecem com mask existente em `src/utils/masks.js` (UX aceitável, sem fricção reportada). Wrapper `<InputBR>` cobre só as 2 categorias com risco de fricção real (data BR `DD/MM/AAAA` e valor monetário `R$ 1.234,56`).

**Why:** reduz risco de regressão (varredura ~24 inputs em vez de ~53) sem perder valor central. Mask existente em CPF/CNPJ/CEP/tel já remove caracteres não-numéricos no `onChange` — paste de "123.456.789-01" funciona aceitavelmente. Phase futura pode adicionar wrapper se usuário reportar fricção real.
**How to apply:** `src/utils/parseInputBR.js` exporta APENAS 2 parsers (`parseBRDate`, `parseBRValue`). Wrapper `<InputBR type="date|value" .../>` intercepta `onPaste` + `onBlur`, delega tudo mais ao `<input>` nativo. CPF/CNPJ/CEP/telefone NÃO entram no wrapper — continuam usando `maskCPF`/`maskCNPJ`/`maskTel`/`maskCEP` direto em `onChange`.

### D-pre-4 — Schema soft delete: `deleted_at TIMESTAMPTZ NULL`

NÃO é boolean `ativo`. Padrão moderno, permite ordenar por data de exclusão e restaurar via SQL.

**Why:** boolean perde informação temporal. TIMESTAMPTZ é convenção em projetos Supabase/Postgres modernos. Migration aditiva (não-destrutiva).
**How to apply:** migration `003_devedores_soft_delete.sql`:
```sql
ALTER TABLE devedores ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ;
CREATE INDEX IF NOT EXISTS devedores_deleted_at_idx ON devedores (deleted_at) WHERE deleted_at IS NULL;
SELECT pg_notify('pgrst', 'reload schema');
```
Index parcial `WHERE deleted_at IS NULL` otimiza queries de listagem (caminho quente).

### D-pre-5 — SEM UI de "Devedores Inativos" nesta phase

**Why:** YAGNI. Banco minúsculo (6 devedores), restauração é evento raro. Adicionar tela de inativos sem demanda real é scope creep.
**How to apply:** Phase futura 7.14b ou 7.15 pode adicionar UI de restauração se demanda surgir. Backlog parking lot.

**How to apply:**

```js
  // "10/05/2026" → "2026-05-10"; "2026-05-10" → "2026-05-10"; senão null
}
export function parseBRValue(input) {
  // "R$ 1.234,56" → 1234.56; "1.234,56" → 1234.56; "1234.56" → 1234.56; senão null
}
```

```jsx
// src/components/ui/InputBR.jsx
export default function InputBR({ type, value, onChange, ...rest }) {
  // type: "date" | "value"  (apenas 2 — D-pre-3)

```

CPF/CNPJ/CEP/telefone NÃO entram no wrapper — continuam usando mask existente em `src/utils/masks.js` direto em `onChange`. Out-of-scope explícito (ver tabela).
