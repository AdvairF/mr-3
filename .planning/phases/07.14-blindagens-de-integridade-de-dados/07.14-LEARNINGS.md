## Decisions (12 D-pre lockeds + 3 ajustes finais)

### D-pre-1 — Soft delete escopo MÍNIMO: devedores apenas
**Why:** usuário só reportou bug bloqueante em pessoas/devedores. Credores hard delete continua funcionando (sem FK violations conhecidas). Phase futura se aparecer demanda real.
**How applied:** todas alterações de schema + filtros aplicam-se SOMENTE a `devedores`. Não tocou outras tabelas.

### D-pre-2 — Validação contrato: SOMENTE criação NOVA, legacy preservado
**Why:** usuário não quer migration agressiva NOT NULL no schema (zero órfãos confirmados, mas evita risco). Validação app-level é reversível, schema NOT NULL não é.
**How applied:** schema mantém `contratos_dividas.devedor_id` e `credor_id` nullable. Validação ocorre em (a) UI `NovoContrato.handleSalvar` e (b) service-layer `criarContrato`.

### D-pre-3 (REVISTA) — Paste-friendly BR escopo: APENAS date + value via `<InputBR>`
**Why:** reduz risco de regressão (varredura ~24 inputs em vez de ~53) sem perder valor central. Mask existente em CPF/CNPJ/CEP/tel já remove caracteres não-numéricos no `onChange` — paste de "123.456.789-01" funciona aceitavelmente. Phase futura pode adicionar wrapper se usuário reportar fricção real.
**How applied:** `parseInputBR.js` exporta APENAS 2 parsers (`parseBRDate`, `parseBRValue`). Wrapper `<InputBR type="date|value" .../>` intercepta `onPaste` + `onBlur`. CPF/CNPJ/CEP/telefone NÃO entram no wrapper — continuam usando `maskCPF/CNPJ/Tel/CEP` direto em `onChange`.

### D-pre-4 — Schema soft delete: `deleted_at TIMESTAMPTZ NULL`
**Why:** boolean perde informação temporal. TIMESTAMPTZ é convenção em projetos Supabase/Postgres modernos. Migration aditiva (não-destrutiva).
**How applied:** Migration 003 com `ALTER TABLE devedores ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ;` + partial index `WHERE deleted_at IS NULL` + `pg_notify('pgrst', 'reload schema')`. Aplicada manualmente em prod 2026-04-27.

### D-pre-5 — SEM UI de "Devedores Inativos" nesta phase
**Why:** YAGNI. Banco minúsculo (6 devedores), restauração é evento raro. Adicionar tela de inativos sem demanda real é scope creep.
**How applied:** restauração manual via SQL (`UPDATE devedores SET deleted_at=NULL WHERE id=X`). Phase futura 7.14b ou 7.15 pode adicionar UI de restauração se demanda surgir.

### D-pre-6 (REVISTA + GREP CONFIRMADO) — Botão "+ Nova Dívida" removido SEM backfill
**Why:** SQL prod confirmou `dividas.contrato_id IS NULL = 0`. Zero órfãos. Backfill avulsas não é necessário.
**How applied:** removido botão `ModuloDividas.jsx:58` + view `nova` + handlers + import. DELETADO arquivo `NovaDivida.jsx` (consumer único era ModuloDividas). PRESERVADO `DividaForm.jsx` (App.jsx:3947 ainda usa — D-pre-12). DEFERRED `ALTER TABLE dividas ALTER COLUMN contrato_id SET NOT NULL` para phase futura.

## Lessons (7 lições centrais transferíveis)

### 1. Soft delete BYPASSA todas as FKs (D-pre-1)
**Contexto:** bug bloqueante prod `eventos_andamento_devedor_id_fkey` (FK NOT CASCADE). 12+ FKs apontam pra devedores, fixar uma a uma seria caro e arriscado.
**Lição:** soft delete via timestamp column não há DELETE físico, então nenhuma FK constraint é violada. Pattern aplicável a outros domínios com FK chains complexas.
**Onde aplicar:** qualquer entidade com >5 FKs apontando pra ela onde DELETE físico é raro mas histórico forense importa.

### 2. Audit grep pré-insert revela consumers ocultos (D-pre-12)
**Contexto:** briefing original tinha `NovaDivida.jsx` como "consumer único do botão removido". Grep audit em sessão de discuss (2026-04-27) encontrou consumer adicional `DividaForm` em `App.jsx:3947` (drawer Pessoas).
**Lição:** sempre executar `grep -rn 'import.*X|<X' src/` ANTES de propor DELETE arquivos. Consumers ocultos em arquivos grandes (App.jsx 8000+ lines) são fonte comum de surprise.
**Onde aplicar:** qualquer phase que propõe deletar arquivo .jsx/.js shared.

### 3. Schema completeness via SQL real, não migrations locais (D-pre-6)
**Contexto:** briefing original presumia `dividas.contrato_id IS NULL` orphans existirem. SQL prod confirmou zero órfãos.
**Lição:** investigar prod data via SQL real ANTES de propor backfill ou ALTER TABLE. Migrations locais podem divergir de prod (executadas manualmente em SQL editor sem commit).
**Onde aplicar:** qualquer phase com `ALTER TABLE` ou data backfill — SEMPRE rodar SQL real em prod ou staging antes.

### 4. Comentários audit trail vs strict-mode shields (D-pre-11)
**Contexto:** D-pre-11 forense exception requer comentário explicativo em processosJudiciais.js mencionando literal `deleted_at=is.null` (token que está sendo NÃO adicionado). Shield strict-mode substring grep flag false positive em comentário.
**Lição:** quando o comentário PRECISA mencionar literal o token que está sendo evitado (audit trail), use shield regex anchored ao callsite real (ex: JOIN string), não substring genérico. Comentário audit trail tem precedence sobre shield strict.
**Onde aplicar:** futuras phases com exceções intencionais que precisam ser documentadas inline.

### 5. Strategy A (shared wrapper delegation) > Strategy B (direct edits) para varreduras amplas
**Contexto:** Plan 04 paste-friendly BR — 35 callsites em 9 components. Strategy A: atualizar shared `<Inp>` para delegar a `<InputBR>`. Strategy B: edit direto em cada callsite.
**Lição:** atualizar shared wrapper internalmente cobre callsites consumers automaticamente sem touchar component files. Pattern DRY que previne migration drift e protege D-pre-7 cumulative invariant (DetalheContrato.jsx intocado).
**Onde aplicar:** qualquer varredura ampla de UI behavior change — sempre auditar shared wrappers (`<Inp>`, `<Btn>`, etc) ANTES de listar direct callsites.

### 6. Encargos motor parity via STRING preservation (Plan 04)
**Contexto:** `<InputBR type="value">` poderia converter para number, mas `parseFloat()` consumers em `devedorCalc.js` esperam string. Conversão number → string seria desnecessária e arriscada.
**Lição:** ao adicionar wrappers em fluxos com motor de cálculo downstream, sempre preservar tipo do dado original (string/number/etc). Test:regressao prebuild hook adiciona camada extra de proteção.
**Onde aplicar:** wrappers que mediam input UI e motor crítico — preservar tipo original + gate prebuild com test:regressao.

### 7. Bugs pré-existentes em features dependentes bloqueiam UAT visual (Plan 01)
**Contexto:** Phase 7.10.bug typo `[object Object]` em ProcessosJudiciais cadastro impediu validação visual SC5 (badge forense) + SC22 (criação NOVA filter). Phases que migram fonte de dados em features dependentes devem audit pré-execute se feature alvo é utilizável em prod.
**Lição:** phases que dependem de features alvo (ProcessosJudiciais, etc) devem executar smoke test pré-execute para confirmar feature alvo funciona em prod. Se não funcionar, deferir SCs específicos para shields automatizados + flag para phase futura que fixe a feature.
**Onde aplicar:** registrada em feedback `proc_judiciais_create_bug_preexisting` — pre-check antes de UAT visual cross-feature.

## Patterns (5 patterns reutilizáveis)

### Pattern 1: 4 plans separados (pattern 7.13 herdado)
**Estrutura:** sub-itens com baseline distinto, commits atômicos bisect-able. Cada sub-item tem domínio próprio (schema vs UI vs service vs varredura ampla) → arquivo PLAN.md independente. Reduz surface area de cada review e facilita bisect em caso de regressão.
**Aplicado em:** Phase 7.14 (4 plans), Phase 7.13 (2 plans). Reutilizável em phases com 3+ sub-itens semantically distinct.

### Pattern 2: Soft delete bypass FKs sem CASCADE
**Estrutura:** `ALTER TABLE X ADD COLUMN deleted_at TIMESTAMPTZ NULL` + partial index `WHERE deleted_at IS NULL` + filter `deleted_at=is.null` em N callsites consumers.
**Aplicado em:** Plan 01 (devedores). Reutilizável para qualquer entidade com FK chains complexas onde DELETE é raro.

### Pattern 3: Exceção forense documentada inline (D-pre-11)
**Estrutura:** comentário explicativo no callsite explica O QUE NÃO está sendo adicionado, com referência à decisão (D-pre-11). Audit trail in-code. Shield regex anchored ao callsite real, não substring genérico.
**Aplicado em:** `processosJudiciais.js:71` JOIN forense. Reutilizável para qualquer exceção intencional que viole pattern dominante.

### Pattern 4: Strategy A (shared wrapper delegation)
**Estrutura:** atualizar shared wrapper internalmente para delegar a novo wrapper específico quando type/prop matches. Public API preservada (callsites untouched). Cobre consumers automaticamente sem direct edit.
**Aplicado em:** Plan 04 `<Inp>` → `<InputBR>` para `type="date|number"`. Reutilizável para qualquer varredura ampla de behavior change em UI.

### Pattern 5: Defense in depth UI + service-layer
**Estrutura:** UI valida ANTES do submit (UX), service-layer guarda ANTES do INSERT (integridade). Ambos throwam mensagem semântica. Previne futuros consumers (scripts, tests, novos componentes) de bypass.
**Aplicado em:** Plan 03 criação de contrato. Reutilizável em qualquer criação de entidade com FKs obrigatórios.

## Surprises (4 surpresas durante execução)

### Surprise 1: DividaForm consumer adicional em drawer Pessoas (D-pre-12 NOVA)
**Discovery:** grep audit pré-Plan 02 encontrou consumer adicional de `<DividaForm>` em `App.jsx:3947` (drawer Pessoas/Devedores). Briefing original tinha como "consumer único = ModuloDividas".
**Resolução:** D-pre-12 NOVA criada — preservar callsite como caminho legacy aceitável (não cria órfã, herda contrato_id). 7.14b backlog para removal futuro.
**Lição transferível:** sempre grep audit antes de propor DELETE.

### Surprise 2: NovaCustaModal e DetalheContrato eram `<Inp>` callsites, não bare `<input>`
**Discovery:** Plan 04 listou DetalheContrato.jsx + NovaCustaModal.jsx como direct callsites com path:line. Auditoria do executor revelou eram `<Inp>` callsites cobertos automaticamente via Strategy A (Task 04-04).
**Resolução:** ambos arquivos NÃO modificados em Plan 04. Strategy A delegation cobriu paste-friendly behavior automaticamente. DetalheContrato.jsx D-pre-7 cumulative invariant preservado.
**Lição transferível:** plans devem auditar shared wrappers antes de listar direct callsites.

### Surprise 3: Junction Set lookup herdou consistência sem filtro adicional
**Discovery:** D-pre-9 strategy b previa filtro em junction queries também. Investigação confirmou que junction `devedores_dividas` (App.jsx:8403, App.jsx:2983) NÃO precisa filtro adicional — devedor soft-deleted some via filtro principal de `devedores`, e qualquer JOIN no front-end (Set lookup D-27 pattern) ignora ids ausentes.
**Resolução:** junction queries NÃO recebem filtro direto. Pattern Set lookup herda consistência automaticamente.
**Lição transferível:** lookup patterns O(1) (Set, Map) com strict id matching herdam consistência via filtro upstream.

### Surprise 4: Shield 4 strict-mode flag em comentário audit trail (D-pre-11)
**Discovery:** substring grep `deleted_at=is.null` em processosJudiciais.js retornou 1 match em comentário explicativo (Task 01-06 mandate D-pre-11). Regex anchored do plan (`devedor:devedores\([^)]*\).*deleted_at=is\.null`) PASSOU.
**Resolução:** false positive aceito. Comentário audit trail mantido como-is. Regex anchored é o shield correto.
**Lição transferível:** shields strict-mode substring podem dar false positives em comentários intencionais. Use regex anchored ao callsite real.
