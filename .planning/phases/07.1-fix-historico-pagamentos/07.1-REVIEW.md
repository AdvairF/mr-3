## Warnings

### WR-01: `snap.parcelas_ids` may be null/undefined — `buildParcelasText` gets a crash-safe fallback but the count expression does not

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:877`
**Issue:** In the `isPagamento` block, line 877 evaluates:

```jsx
{snap.parcelas_amortizadas ?? snap.parcelas_ids?.length ?? 0}
```

`snap.parcelas_ids?.length` is safe, but `snap.parcelas_ids` passed to `buildParcelasText` on line 876 is not validated before being passed. `buildParcelasText` itself guards for `!parcelasIds || parcelasIds.length === 0` and returns `"—"` (line 394), so no crash occurs there. However, the inline count expression on line 877 is only safe because of the `?.` — if the RPC ever returns `parcelas_ids` as a non-array (e.g., a number), `.length` would be `undefined` and the `?? 0` would silently produce `0 amortizada(s)` instead of an error. This is a latent data-contract assumption with no runtime guard.

**Fix:** Assert the array type before rendering to make the contract explicit:

```jsx
const parcelasIds = Array.isArray(snap.parcelas_ids) ? snap.parcelas_ids : [];
// ...
<div>
  Parcelas: {buildParcelasText(parcelasIds, dividas)}
  {" "}({snap.parcelas_amortizadas ?? parcelasIds.length} amortizada(s))
</div>
```

### WR-02: `fmtDataHora` drops the time component — label mismatch for same-day events

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:14–21`
**Issue:** `fmtDataHora` is named to imply it formats both date *and* time, but `toLocaleDateString` only outputs the date (e.g., `22/04/2026`). If two events occur on the same day — which is common when a payment is registered and then immediately reverted — both timeline entries will display identical timestamps. The user has no way to distinguish the order from the UI.

**Fix:** Either rename to `fmtData2` / `fmtDataEvento` to remove the false promise, or include time:

```js
function fmtDataHora(iso) {
  if (!iso) return "—";
  try {
    const d = new Date(iso);
    if (isNaN(d.getTime())) return "—";
    return d.toLocaleString("pt-BR", {
      timeZone: "America/Sao_Paulo",
      day: "2-digit", month: "2-digit", year: "numeric",
      hour: "2-digit", minute: "2-digit",
    });
  } catch { return "—"; }
}
```

This would output `22/04/2026, 14:30` and actually justify the "Hora" in the name.

### WR-03: `handleAbrirFormPagamento` has no error handling — a single failing `listarPagamentos` call silently leaves the form inaccessible

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:310–319`
**Issue:** The `for...of` loop at line 314 awaits `listarPagamentos` sequentially with no try/catch. If any call rejects, the unhandled rejection propagates out of `handleAbrirFormPagamento`, `setRegistrandoPagamento(true)` on line 319 is never reached, and the payment form simply does not open. The user sees nothing — no error toast, no feedback.

```js
async function handleAbrirFormPagamento() {
  // ...
  for (const p of parcelasAbertas) {        // line 314
    const pgtos = await listarPagamentos(p.id);  // unhandled rejection
    total += calcularSaldoPorDividaIndividual(p, pgtos, hoje_str);
  }
  setRegistrandoPagamento(true);  // never reached on failure
}
```

**Fix:**

```js
async function handleAbrirFormPagamento() {
  const hoje_str = typeof hoje === "string" ? hoje : hoje.toISOString().slice(0, 10);
  const parcelasAbertas = (dividas || []).filter(d => !d.saldo_quitado);
  let total = 0;
  try {
    for (const p of parcelasAbertas) {
      const pgtos = await listarPagamentos(p.id);
      total += calcularSaldoPorDividaIndividual(p, pgtos, hoje_str);
    }
  } catch (e) {
    toast.error("Erro ao calcular saldo devedor: " + e.message);
    return;
  }
  setSaldoCalculado(total);
  setRegistrandoPagamento(true);
}
```

### WR-04: `historicoCarregado` flag is never reset when `contrato.id` changes — stale history shown for a new contract

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:169–180` and `182–194`
**Issue:** The `useEffect` on lines 169–180 resets `historicoCarregado` to `false` when `contrato.id` changes. However, it does **not** reset `historicoAberto`. The lazy-load guard on line 183 is:

```js
if (!historicoAberto || historicoCarregado) return;
```

If the user had the Histórico section open on Contract A, then navigates to Contract B (triggering the `contrato.id` effect), `historicoAberto` remains `true` after the reset, `historicoCarregado` is now `false`, so the effect on line 182 fires and correctly re-fetches. This is actually fine in isolation.

The actual risk is the inverse: `pagamentosCarregado` is reset on line 178 and `pagamentosAberto` is reset on line 176 — but `historicoAberto` is reset on line 172. The sequencing is correct. On closer inspection this warning is lower-risk than initially apparent, but warrants a unit test because the two parallel accordion sections (`historicoAberto` + `pagamentosAberto`) follow the same lazy pattern and a future refactor could easily break the reset symmetry.

**Fix:** Add a comment documenting the reset dependency to make the intent clear and prevent future regression:

```js
useEffect(() => {
  // Reset ALL accordion state when contract changes.
  // IMPORTANT: both *Aberto and *Carregado flags must be reset together
  // so the lazy-load guards re-fire for the new contract.
  setEditForm(initEditForm(contrato));
  setEditando(false);
  setHistoricoAberto(false);
  setHistorico([]);
  setHistoricoCarregado(false);
  setRegistrandoPagamento(false);
  setPagamentosAberto(false);
  setPagamentosContrato([]);
  setPagamentosCarregado(false);
  setSaldoCalculado(0);
}, [contrato.id]);
```

### IN-01: `diffEntries` rendering guard does not exclude `isPagamento` — dead code path

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:819–825` and `882`
**Issue:** `diffEntries` is computed as `[]` when `isCriacao || isPagamento` (line 819). The render block at line 882 guards on `!isCriacao && diffEntries.length > 0`. This is correct — `isPagamento` events will have `diffEntries = []` so `diffEntries.length > 0` will be `false`. The block is silently skipped. However, the guard could be made explicit for readability:

```jsx
{/* currently: */}
{!isCriacao && diffEntries.length > 0 && ( ... )}

{/* clearer: */}
{!isCriacao && !isPagamento && diffEntries.length > 0 && ( ... )}
```

This makes the three-way branching (`isCriacao` / `isPagamento` / diff) immediately visible without requiring the reader to trace `diffEntries` back to its initializer.

### IN-02: `data` field in payment form has no empty-string guard before date comparison

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:331`
**Issue:** `handleRegistrarPagamento` checks `if (data > hoje_str)` on line 331, but `data` may be `""` (initial state). An empty string compares less than any date string, so the check passes silently — the real guard is the `!valor || valor <= 0` check. However, a missing date should surface a user-visible error rather than fail silently at the API call.

**Fix:**

```js
if (!data) {
  toast.error("Informe a data do pagamento.");
  return;
}
```

Add this before the `data > hoje_str` check.

### IN-03: `buildParcelasText` sorts all `dividas` on every call — called twice per history event

**File:** `src/mr-3/mr-cobrancas/src/components/DetalheContrato.jsx:393–408`
**Issue:** `buildParcelasText` sorts `dividas` on every invocation (line 395). In the historico rendering loop, it is called once per `isPagamento` event (line 876) and once per row in the pagamentos table (line 602). For typical contract sizes this is harmless, but the sort is O(n log n) and the result is identical for every call since `dividas` does not change within the render. Memoizing outside the render loop would be cleaner.

**Fix (optional):** Hoist the sorted array outside the map:

```js
const sortedDividas = [...(dividas || [])].sort((a, b) =>
  (a.data_vencimento || "").localeCompare(b.data_vencimento || "")
);
// Then pass sortedDividas to buildParcelasText and simplify buildParcelasText
// to skip the internal sort.
```

This is a code-quality note only — no correctness impact.
