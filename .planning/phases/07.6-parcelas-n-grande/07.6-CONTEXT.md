## 1. Problema

Formulário de "Novo Documento" (`AdicionarDocumento.jsx:172`) tem `max="360"` no input "Nº de Parcelas". Empréstimos reais do domínio forense excedem 360:

- Consignado: 96 parcelas
- Imobiliário 30 anos: 360 parcelas
- **Imobiliário 35 anos: 420 parcelas** ← bloqueado hoje

Além disso, `TabelaParcelasEditaveis.jsx` renderiza N linhas em `<table>` sem virtualização. Com N=500 seriam ~1500 nós DOM (3 inputs por linha × N) e cada keystroke dispara re-render do array inteiro. Performance degrada antes mesmo de chegar em 999.

## 2. Escopo aprovado

**Fix A** — Bump limite em `AdicionarDocumento.jsx`:
- `max="360"` → `max="999"`
- `podesSalvar`: adicionar `parseInt(numParcelas) <= 999`
- `handleSalvar`: se N<1 ou N>999, toast vermelho bloqueia submit

**Fix B** — Virtualização em `TabelaParcelasEditaveis.jsx`:
- Instalar `@tanstack/react-virtual` no submodule (`src/mr-3/mr-cobrancas/package.json`)
- Refactor do render da tabela pra usar `useVirtualizer`
- Preservar: validações, botão "Sugerir datas", indicador Soma/Total/Δ, readonly em parcelas pagas, contrato de props (D-02 da 7.5)

## 3. Fora do escopo

- Bulk edit de parcelas (mudar todas as datas/valores de uma vez) — phase futura
- Backend/schema — `num_parcelas INT CHECK >= 1` (contratos.js:43) já aceita qualquer N
- Legacy NovaDivida form em `App.jsx:1073,1148` — não tem tabela editável; não se beneficia de virtualização
- Teste unitário de virtualização — projeto não tem jsdom/RTL configurado; UAT manual cobre
- Otimização contra re-render cascade por keystroke — monitorado em UAT, vira pendência futura se detectado

## 4. Decisões locked (discuss 2026-04-23)

- **D-01** — Motor Art.354 (`utils/devedorCalc.js`, `services/pagamentos.js`) **NÃO** é tocado. 9/9 testes regressão TJGO preservados por invariante estrutural. Nenhum teste novo adicionado (phase é UI-only).

- **D-02** — Limite de parcelas = **999**. Teto absoluto prático que cobre 35 anos imobiliário (420). Enforçado em 2 camadas: `max="999"` HTML + validação JS `N >= 1 && N <= 999` com toast vermelho bloqueante. Schema inalterado.

- **D-03** — Biblioteca = **`@tanstack/react-virtual` v3**. Justificativa: hook-based (`useVirtualizer`), permite DOM flexível (CSS grid em vez de forçar `<div>` como `react-window`), ~3-5 KB gzipped, stack TanStack futura (Table/Query já no ecosistema). Preferência registrada pelo usuário, ratificada por análise técnica.

- **D-04** — **Sempre virtualizar**, sem threshold. Mesmo código pra N=3 e N=500. Simplicidade (um path) e consistência visual (não há "pulo" no render acima de N=50). Spec original do usuário dizia `> 50 virtualiza, senão normal`; reconsiderada em favor de simplicidade.

- **D-05** — Estrutura DOM = **CSS grid com role-attrs** (`role="table"`, `role="row"`, `role="cell"`). NÃO usar `<table>` + spacer-row com colspan. Acessibilidade preservada via ARIA roles; elimina hack de absolute positioning dentro de `<tbody>` (browsers normalizam agressivamente).

- **D-06** — Container de scroll vertical = **`maxHeight: 400px` fixo**. Ponto de partida. Se tela grande ficar pequeno, ajuste em phase futura. `overflowY: "auto"` no container.

- **D-07** — `overscan = 10` linhas no `useVirtualizer`. Mitiga perda de foco durante scroll quando usuário digita e rola. Não elimina 100%; UAT valida severidade.

- **D-08** — Validações existentes 100% preservadas:
  - Soma das parcelas === `valorTotal` (tolerância R$ 0,01)
  - Todas as datas preenchidas
  - Ordem não-decrescente por `numero`
  - Readonly em valor+data de parcelas com `pagamentos_divida` OU `saldo_quitado=true`
  - Indicador "Soma / Total / Δ" em tempo real SEM regressão (cor verde quando `|diff| <= 0,01`, vermelha caso contrário)

- **D-09** — `sugerirDatas` com N grande permanece **síncrono**. `.map` com `new Date() + setDate()` pra N=500 ≈ 10ms. Async não necessário. Preocupação inicial do usuário foi validada como infundada na investigação.

- **D-10** — Re-render cascade por keystroke é **risco conhecido e não mitigado nesta phase**. `React.memo` NÃO aplicado preventivamente. UAT com N=100 monitora lag perceptível. Se detectado, abre pendência pra phase futura (Phase 7.6.1 hipotética ou inclusão em 7.8 tech debt). **NÃO bloqueia 7.6**.

- **D-11** — UAT manual obrigatório (9 testes regressão não cobrem UI). Cenários mínimos:
  - **SC-1**: criar documento N=10 (path virtualizado com poucas linhas — sanity)
  - **SC-2**: criar documento N=100 (path virtualizado realista) — observar lag de keystroke
  - **SC-3**: Soma/Total/Δ atualiza em tempo real scrollando + editando parcelas fora da viewport
  - **SC-4**: "Sugerir datas" mensal com N=100 — verificar que datas fora da viewport foram preenchidas
  - **SC-5**: tentar N=1000 — toast vermelho bloqueia, sem submit
  - **SC-6**: foco retention — digitar em parcela 50, scrollar até topo, voltar: foco no input esperado
  - N=500/999 assumido escala se N=100 funcionar

- **D-12** — Phase 7.5.2 backlog (encargos possivelmente não propagados em `adicionarDocumento` com `parcelasCustom`, registrado em memory) é pendência PRÉ-EXISTENTE. Se UAT de 7.6 parecer quebrado nesse ponto, é regressão de 7.5, não de 7.6. Flag no UAT, não bloqueio.

## 6. Achados da investigação (pré-planning)

### Limite de 9 era falso

Memory do usuário dizia "provavelmente tem `max='9'`". A investigação em `AdicionarDocumento.jsx:172` mostrou `max="360"` — limite já foi bumpado em edit anterior não documentado. Ainda bloqueia 420, portanto phase segue justificada.

### Pre-push hook não bloqueia package.json

`.git/hooks/pre-push` só valida bump do submodule `src/mr-3`. Diff de arquivos não é inspecionado. Instalação de `@tanstack/react-virtual` (`package.json` + `package-lock.json`) passa normalmente. Preocupação #1 do usuário validada como infundada.

### Perf de "Sugerir datas" com N grande

`.map` síncrono com `new Date()` + `setDate()` pra N=500 ≈ 10ms. Não precisa async. Preocupação #4 do usuário validada como infundada.

### Edge cases flagged

- **E1 re-render cascade**: cada keystroke faz `.map` sobre `parcelas` e chama `onChange(next)` no pai → rerender. Virtualização cuida do DOM mas React reconciler ainda caminha array. **D-10** cobre monitoramento em UAT.
- **E2 foco perdido ao scrollar**: linha desmonta ao sair da viewport; foco vai pro body. **D-07** mitiga com `overscan=10`.
- **E3 container vertical**: tabela hoje não tem `maxHeight`. **D-06** define 400px.
- **E4 inconsistência threshold**: evitada pela **D-04** (sempre virtualizar).
