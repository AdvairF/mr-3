## Decisions

### D-01 — Motor Art.354 intocado com dual gate (C1 pré-commit + C2 pós-commit)
Motor `devedorCalc.js` + services `pagamentos.js`, `dividas.js` permaneceram com zero deltas. Validado por DOIS gates nomeados: C1 antes de `git add` (working tree check), C2 após commit (commit content check).

**Rationale:** invariante preservado desde 7.8 (feat saldo atualizado) e 7.8.1 (fix filter cross-contrato). Dual gate captura tanto edição acidental quanto drift de index (arquivo staged antes de C1).
**Source:** 07.8.2a-CONTEXT.md D-01; 07.8.2a-01-PLAN.md Task 8 Gates C1 + C2

### D-05 — Callers completude (6 handlers com invalidate/remove) antes de toast.success
Todos os 6 handlers que mutam estado (`handleRegistrarPagamento`, `handleExcluirPagamento`, `handleSalvarParcelasCustom`, `handleExcluirDocumento`, `handleExcluirContrato`, `handleSalvar` em AdicionarDocumento) DEVEM chamar `invalidateContrato(contrato.id)` ou `removeContrato(contrato.id)` entre a service call e o `toast.success`/`setState`.

**Rationale:** blinda silent-stale class — mutação que não invalida deixa cache desatualizado. Lição direta do bug cross-contrato 7.8.1 (Mendes e Mendes exibia pagamentos do TRADIO). Shield grep com report format obrigatório reproduz tabela 6-linhas no SUMMARY.
**Source:** 07.8.2a-CONTEXT.md D-05 NOVA (lição 7.8); 07.8.2a-01-PLAN.md Task 8 Gate E

### D-06 — Fingerprint deep 12 campos verbatim (captura fallback chain do motor)
Fingerprint inclui `id, valor_total, data_vencimento, data_origem, data_inicio_atualizacao, indexador, multa_pct, juros_tipo, juros_am, honorarios_pct, art523_opcao, saldo_quitado` — 12 campos na ordem exata.

**Rationale:** `data_inicio_atualizacao || data_vencimento || data_origem` é o fallback do motor (devedorCalc.js:387); fingerprint precisa capturar os 3 senão drift silencioso. Custo ~2-4ms pra 200 contratos é aceitável.
**Source:** 07.8.2a-CONTEXT.md D-06; 07.8.2a-01-PLAN.md Task 1

### D-11 — Kickoff eager (batches de 10 via requestIdleCallback) em vez de lazy
Cache populado pra TODOS contratos no mount em batches de 10, não lazy por cell.

**Rationale:** lazy criaria lista "pulando" (UX ruim); pré-requisito da 7.8.2b (ordenação); alinha com gatilho 1 da SWR. Custo 200-400ms diluído pelo idle callback.
**Source:** 07.8.2a-CONTEXT.md D-11

### D-14 — 3 PAUSAs humanas proporcionais à superfície de risco (7 arquivos)
PAUSA #1 UAT SC1-SC8 → PAUSA #2 diff consolidado pós-commit submódulo → PAUSA #3 bump review pai pré-push. Todas bloqueiam passo seguinte sem OK explícito.

**Rationale:** proporcional ao escopo — phase com 7 arquivos novos/modificados + 6 handlers patched + estado global (cache singleton) justifica 3 gates humanos. Proven value: PAUSA #1 pegou H3 que 7 gates CI passaram.
**Source:** 07.8.2a-CONTEXT.md D-14; 07.8.2a-02-PLAN.md Tasks 1/2/4

### SC1 Amend (pós-checker) — 5→7 exports (2 helpers de observabilidade)
CONTEXT.md §5 SC1 alterado durante revision loop do plan-checker para listar 7 exports: 5 API principal (`useSaldoContrato`, `invalidateContrato`, `invalidateAll`, `removeContrato`, `kickoffBatch`) + 2 helpers (`subscribeToCache`, `getLastCalculadoEm`).

**Rationale:** SC5 (timestamp reativo no header) exige subscrição ao cache. Alternativas (polling setInterval, hook separado que ainda precisa exportar funções internas) foram piores. Commit separado `docs(07.8.2a): amend SC1 — add observability exports` (9cceb02) preservou audit trail da decisão.
**Source:** commit 9cceb02; 07.8.2a-CONTEXT.md §5 SC1 (atual)

## Lessons

### H3 — Adapter espera string "YYYY-MM-DD"; Date object lexicographically coerces wrong
`calcularDetalheEncargosContrato(dividas, pagamentos, hoje)` espera `hoje` como string per JSDoc (devedorCalc.js:846). Executor passou `new Date()` por entender errado o comentário D-09 do CONTEXT. Resultado: motor comparava `divida.data_vencimento < stringified Date` lexicograficamente → dias_atraso = NaN → saldo retornava nominal (R$ 12.000 em vez de R$ 36.652,56).

**Context:** descoberto SÓ em UAT manual (PAUSA #1, SC1) visualmente comparando listagem vs detalhe do mesmo contrato. Todos os 7 gates CI passaram: regressão 14/14, build 0, D-01/D-02/D-05/F todos OK. Fix: trocar `new Date()` por parâmetro `hoje` que já chegava correto (1 linha).
**Source:** 07.8.2a-01-SUMMARY.md seção "Gate C2 reformulado pós-fact"; commit 608c124

### Gap estrutural — shields testando infraestrutura de cache NÃO pegam bugs de integração com motor
Os 3 shields originais (cross-entity, virada dia, invalidate isolado) cobriam mecânica do cache (Map pub/sub, fingerprint reatividade, delete isolado). Nenhum testava equivalência `cache.saldo === adapterDireto(input).saldoAtualizado`. Por isso H3 passou CI. Shield 5 novo (equivalence test, gold test) teria pego o bug.

**Context:** lição lampada em `memory/feedback_shields_wrapper_must_test_equivalence.md`. Aplicável a qualquer wrapper/cache/memoization de função pura subjacente.
**Source:** 07.8.2a-01-SUMMARY.md §Deviations + seção "Gate C2 reformulado"; commit 608c124 (shields 4+5)

### Gate C2 falso positivo — `git log -1 --stat` casa em commit message rica
Shield C2 original usava `git log -1 --stat | grep -qE "devedorCalc|services/pagamentos|services/dividas"`. Mensagem do commit 608c124 referenciava arquivos do motor pra auditar causa raiz H3 → grep casou em COMMIT MESSAGE, não em diffstat real. FAIL falso positivo disparou Rule 3 → paramos corretamente.

**Context:** fix aplicado ao 01-PLAN.md (pre-Task-5 bundle): `--stat` → `--format="" --name-only`. Preserva commit 608c124 intacto (evidência forense). Lição em `memory/feedback_gate_motor_grep_use_name_only.md`.
**Source:** conversational log 2026-04-24 PAUSA #1; 07.8.2a-01-SUMMARY.md seção "Gate C2 reformulado pós-fact"

### Grep shield fraco casa em prose PT-BR (não só em código)
Shield D-02 isolation `grep -nE "import.*DetalheContrato|import.*AdicionarDocumento"` casou no comentário PT-BR `"NÃO importa DetalheContrato"` no header do hook (verbo português "importa" → prefixo "import"). Shield D-10 display-only casou em `onClick`, `role="button"`, `cursor: pointer` também em comentário. Executor reescreveu comentários pra satisfazer grep (D-2 deviation).

**Context:** lição em `memory/feedback_grep_shield_strict_imports.md`. Fix futuro: grep `^import .* from ["']...["']` em vez de substring livre. Não aplicável retroativamente a shields já locked em phases em execução.
**Source:** 07.8.2a-01-SUMMARY.md §Deviations #2

### Arquitetura UI/state split ModuloContratos vs TabelaContratos — planner cego
ModuloContratos.jsx = wrapper de estado (dividas, pagamentos, total_pago, saldo_restante, filtros, kickoffBatch call, botão 🔄, timestamp); TabelaContratos.jsx = HTML puro (thead/tbody/tr/td, paginação, hover). Planner assumiu que `<table>` estava inline em ModuloContratos.

**Context:** descoberto em Task 4 execução. Fix: prop opcional + deviation D-1 documentada. Backlog 7.8.2b/c pode considerar consolidar OU renomear pra clareza (`ModuloContratosContainer` + `ContratosTable`).
**Source:** 07.8.2a-01-SUMMARY.md §Deviations #1 (discovery)

### Windows Git Bash UTF-8 multi-byte — grep nativo falha em emoji
`grep -c "🔄 Atualizar"` com grep nativo do Git Bash retorna 0 apesar de string estar presente (4-byte UTF-8 `f0 9f 94 84` não é processado). Ripgrep (`rg -c`) encontra corretamente.

**Context:** observacional, não bug. Em acceptance criteria futuras que envolvam emojis, especificar `rg` em vez de `grep`, ou usar hexdump como fallback.
**Source:** 07.8.2a-01-SUMMARY.md §Deviations #3

## Patterns

### Shield equivalence test — `wrapper(input).toEqual(adapterDireto(input))`
Gold test pra qualquer wrapper/cache/memoization. Testa que wrapper NÃO degrada o cálculo da função subjacente. Shield 5 da 7.8.2a teria pego H3 em CI antes do commit.

**When to use:** sempre que phase introduz camada sobre função pura (cache SWR, memoization, adapter thin, hook wrapping util). Complementa — não substitui — shields de propriedade observável (Shield 4 "saldo > nominal" é útil mas não suficiente).
**Source:** 07.8.2a commit 608c124 (Shield 5); memory/feedback_shields_wrapper_must_test_equivalence.md

### Gate file-intocado — `git log -1 --format="" --name-only | grep -qE "..."`
Shield pra validar "arquivo X não foi modificado neste commit". `--format=""` suprime header + mensagem; `--name-only` lista só paths. Combinado: grep casa EXCLUSIVAMENTE em arquivos tocados, zero risco de falso positivo em commit message rica.

**When to use:** todo shield de "motor intocado", "arquivo legado não regredido", "whitelist de paths tocados" etc. Evita o anti-pattern de `git log -1 --stat` que mistura message + stat.
**Source:** 07.8.2a-01-PLAN.md Gate C2 reformulado (L1002/1006/1030/1077); memory/feedback_gate_motor_grep_use_name_only.md

### Dual gate pré-commit + pós-commit para invariante de arquivos intocados
Gate C1 roda antes de `git add` (checka working tree), Gate C2 roda após `git commit` (checka commit content). Captura tanto edição acidental no tree quanto drift de index (arquivo staged antes de C1).

**When to use:** qualquer invariante onde "esse arquivo NÃO pode aparecer no commit". Cobertura dupla é proporcional ao risco (ex: motor Art.354 em phase que não deveria tocá-lo).
**Source:** 07.8.2a-01-PLAN.md Task 8 Gates C1/C2

### 3 PAUSAs humanas proporcionais — UAT → diff submódulo → bump review pai
Phase com superfície grande (7+ arquivos) + estado global (cache singleton) + irreversibilidade (push prod) merece 3 gates humanos sequenciais:
1. UAT local (pega bugs comportamentais)
2. Diff consolidado pós-commit submódulo (pega push submódulo errado)
3. Bump review pai (pega bump com SHA errado)

**When to use:** qualquer phase com ≥2 camadas (submódulo + pai) + superfície ≥7 arquivos + estado mutável global. Proved value em 7.8.2a: PAUSA #1 pegou H3 que CI não pegou.
**Source:** 07.8.2a-CONTEXT.md D-14; 07.8.2a-02-PLAN.md

### Commit message rica como evidência forense — não mutilar pra passar shield
Quando shield falha por falso positivo causado por commit message, a resposta correta é fix no shield (ou override humano), NÃO amend da mensagem. Mensagem rica é evidência pra debugger futuro; "mudar mensagem só pra passar CI" vira rubber-stamp de CI.

**When to use:** sempre. Mensagens de commit são contratos históricos. Shield defeituoso = fix do shield, não amend do commit.
**Source:** decisão conversacional 2026-04-24 pós-commit 608c124

### Grep preventivo no acceptance criteria pós-bug
Após fix de bug de integração (tipo H3), adicionar grep pattern do ANTI-PADRÃO ao acceptance criteria das phases futuras: `grep -nE "calcularDetalheEncargosContrato\(.*new Date\(\)"` → 0 matches. Catches recurrence do mesmo bug em phase posterior.

**When to use:** após qualquer fix de bug recorrente-possível em pattern identificável. Trivial de adicionar, zero custo de CI, pega regressão de semana/mês futura.
**Source:** commit 608c124 body; memory/feedback_shields_wrapper_must_test_equivalence.md

### Option Z — defer docs commit pra Task 5 bundle em vez de "fix commit separado"
Quando descoberta factual (PLAN.md nunca committado no pai) invalida instrução literal do usuário ("commit separado de fix"), clarify + oferecer alternativas em vez de silent inventar solução ou stash-kabuki. Z preserva convenção do repo (bundles docs em Task 5) + preserva audit via seção explícita no SUMMARY.

**When to use:** qualquer divergência entre instrução humana e realidade git. Clarify > silent fix > commit kabuki.
**Source:** decisão conversacional 2026-04-24 PAUSA #1 pós-fix

## Surprises

### UAT visual pegou H3 que 7 gates CI NÃO pegaram
Regressão 14/14, build 0, motor diff C1/C2 ambos OK, isolation, shield D-05 6/6 YES, clean status — todos PASS. Bug comportamental (listagem R$12k vs detalhe R$36.6k) só apareceu em UAT manual comparando 2 views do mesmo contrato.

**Impact:** valida o modelo das 3 PAUSAs humanas como gate NÃO redundante com CI. PAUSA #1 não é ceremonial — captura bugs de integração/percepção que greps sintáticos não detectam.
**Source:** UAT conversacional 2026-04-24 Mendes e Mendes

### Plan-checker NÃO pegou H3 antes da execução
Plan-checker aprovou verde (0 blockers, 0 warnings após revision loop). Ainda assim H3 passou. Porque plan-checker avalia estrutura do plan (frontmatter, tasks têm read_first, acceptance criteria têm grep), não semântica do que o plan produz em runtime.

**Impact:** scope do plan-checker é estrutural, não funcional. Não confiar em checker verde = 100% phase correta. PAUSA humana é necessária. Regressão suite precisa de equivalence tests pra cobrir semântica.
**Source:** plan-checker verification 2026-04-24; commit 608c124

### Gate C2 falso positivo — Rule 3 comportou corretamente
Shield disparou FAIL (falso positivo em commit message). Eu parei automaticamente e surfaced pra usuário sem auto-reset. Usuário aprovou Option A+C (preservar commit, fix shield). Padrão de "halt + report + human decide" funcionou exatamente como Rule 3 prescreveu.

**Impact:** hard rules do usuário ("STOP on any gate failure") não são só burocracia — são protocolo de safety que evita auto-reset/auto-amend destructive. Rule 3 provou seu valor na primeira trigger.
**Source:** conversational log 2026-04-24 Gate C2 FAIL report

### TabelaContratos.jsx discovery — 9º arquivo necessário, não previsto
Plan.files_modified listava 8 arquivos. Na execução, Task 4 descobriu que a `<table>` realmente fica em TabelaContratos.jsx (não ModuloContratos). Executor adicionou 9º arquivo com prop opcional backward-compat.

**Impact:** planner não tem acesso a leitura profunda da codebase que executor tem. Deviations tipo "9º arquivo necessário" são inevitáveis. Executor com hard rules (Rule 2: STOP se gate falhar) mitiga risco. Documentar em SUMMARY §Deviations + discovery note para backlog.
**Source:** 07.8.2a-01-SUMMARY.md §Deviations #1

### SC8 timestamp HH:MM — falso-negativo em testes sub-minuto
Re-UAT SC8 primeiro parecia FALHAR porque clique em "🔄 Atualizar" não mudava timestamp visível (HH:MM sem segundos → clicks dentro do mesmo minuto aparecem idênticos). Aguardar 2 minutos confirmou que timestamp ATUALIZA, só falta granularidade visual.

**Impact:** registrado como UX refinement no backlog v1.4 (memory/project_v14_backlog_post_76.md). Não é bug da 7.8.2a — é polish pra 7.8.2c ou phase equivalente. Considerar HH:MM:SS OU indicador relativo "há Xs".
**Source:** UAT conversacional 2026-04-24 pós-fix H3

### Regressão 14/14 → 16/16 permanente (shields 4 e 5 aditivos)
O bug H3 forçou adicionar 2 shields de regressão permanente (property + equivalence). Nenhum shield foi removido. Custo ~0ms por run; cobertura aumenta N→N+2 em perpetuidade.

**Impact:** cada bug descoberto pós-CI é oportunidade de adicionar shield que blinda recurrence. Resistir à tentação de apenas fixar o código sem adicionar teste.
**Source:** commit 608c124 (package.json test:regressao continua apontando 2 arquivos; 16 tests total)

### Gate C1/C2 split pré/pós-commit provou valor imediato
Revision pré-execução adicionou split Gate C1 (pré-commit) + C2 (pós-commit). Executava parece over-engineering. Mas na execução real, C2 disparou falso positivo e C1 passou OK — split permitiu diagnóstico preciso (working tree OK, commit content ambíguo → culpa do grep, não do commit).

**Impact:** gates nomeados e segregados por timing permitem diagnóstico granular quando um falha. Shield unificado "motor intocado" teria dado apenas FAIL, sem pista de onde.
**Source:** commit 608c124 gate execution log; 07.8.2a-01-SUMMARY.md

### Commit 608c124 preservado — decisão humana sobre audit trail
Quando Gate C2 disparou FAIL, opção óbvia era amend commit pra remover referências ao motor. Usuário recusou explicitamente ("mutilar commit pra passar shield defeituoso é anti-pattern"). Decisão reveals mindset: commit history é mais valioso que CI perfeito.

**Impact:** precedente pra phases futuras: quando houver conflito entre CI pass e audit trail, priorizar audit. CI é ferramenta; commit history é o registro.
**Source:** decisão conversacional 2026-04-24 Gate C2 FAIL report
