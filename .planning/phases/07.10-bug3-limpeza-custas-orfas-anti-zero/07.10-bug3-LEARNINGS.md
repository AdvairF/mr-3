## Lições transferíveis (5)

### Lição 1 — ABORT GATE durante execução pode reduzir escopo (NOVA)

**Sintoma:** ABORT GATE 6.b (D-pre-5) rodou após Tasks 1-5 commitadas + pushadas + Vercel deploy Ready. Detectou divergência: Q14 reportou `regulares_sem_contrato=0` (esperado: 3, advair). Investigação granular Q B preview revelou que 3 advair têm `_so_custas=true` + `qtd_custas=1` cada (custas reais R$ 150 cada) — **NÃO são órfãs vazias**.

**Decisão pragmática:** scope reduction (Sub-cleanup B → Phase 7.10.bug5 BACKLOG) preserva trabalho cumulative das Tasks 1-5. Alternativa ao ABORT total quando descoberta é de **escopo**, não de **mecânica**.

**Lição:** ABORT GATE não é binário (PASS/ABORT total). Quando descoberta é:
- **Mecânica** (premissa arquitetural invertida — caso 7.13c): ABORT phase, REJECT, replan
- **Escopo** (sub-item da phase é fora do escopo real): scope reduction in-place, defer sub-item pra phase nova BACKLOG

**Memory feedback NOVO:** `abort_gate_during_execution_can_reduce_scope`

**Aplicação:** Quando ABORT GATE detecta divergência pré-DELETE/UAT, primeiro pergunta: "isso muda mecânica do que já foi feito ou apenas o escopo do que falta?" — se for escopo, in-place adjustment + phase nova BACKLOG. Lição complementar a `feedback_post_hoc_d_pre_must_be_validated_in_child_phase`.

### Lição 2 — Helpers com prefix interno exigem audit de callsites preexistentes (NOVA)

**Sintoma:** Task 2 do bugfix introduziu `dbDelete("dividas", \`id=eq.${encodeURIComponent(dividaId)}\`)`. Sintaxe JavaScript válida, build PASS, test:regressao 34/34 PASS, Gate D-01 dual PASS, Vercel deploy PASS, console limpo no carregamento. **Bug semântico** descoberto **APENAS em UAT humano SC3** (operador clicou Excluir): PostgREST 400 `invalid input syntax for type uuid: 'id=eq.<UUID>'`.

**Causa raiz:** `dbDelete` em `supabase.js:76` é definido como `(t, id) => sb(t, "DELETE", null, "?id=eq.${id}")` — **constrói o filtro internamente**. Espera UUID puro, NÃO query string completa.

Quem escreveu Task 2 inferiu padrão do helper vizinho `dbGet` (L73): `(t, q="") => sb(t, "GET", null, q ? "?${q}" : "?order=id.asc")` — `dbGet` aceita query string completa. **Helpers com nomes similares têm semântica diferente.**

Os 2 callsites preexistentes em `services/contratos.js` que funcionam em prod cumulative há meses passam UUID puro:
- L393 `excluirContrato`: `dbDelete(TABLE, contratoId)` ✓
- L471 `excluirDocumento`: `dbDelete("documentos_contrato", documentoId)` ✓

**Lição:** Phases que modificam comportamento de helper compartilhado (dbDelete, dbUpdate, dbInsert) devem incluir Task pré-execute:
1. Grep callsites preexistentes do helper (`grep -n "dbDelete(" src/`)
2. Auditar 2-3 amostras pra padronizar formato do segundo arg
3. Validar que assinatura nova bate com padrão preexistente

**Memory feedback NOVO:** `helpers_with_internal_prefix_require_callsite_audit`

**Aplicação:** Antes de adicionar callsite novo de helper compartilhado, executar grep pré-execute pra padronizar com preexistentes. Especialmente crítico pra helpers com prefix interno (dbDelete, dbUpdate) — a inconsistência de API dentro do mesmo módulo (`dbGet` aceita filter completo, `dbDelete` aceita só ID) é armadilha clássica.

### Lição 3 — Validadores client preexistentes interceptam novos guards em phases anti-X (NOVA)

**Sintoma:** Phase 7.10.bug3 D-pre-3 prometia "anti-zero validation server+client mensagem unificada" — Task 3 (server `criarCusta` throw) + Task 4 (client `handleCriarCusta` em DetalheContrato). Plan assumia 2 validadores.

**Descoberta SC4:** Form de cadastro de custa (preexistente Phase 7.9 ou anterior) JÁ TINHA validador client com mensagem própria: `"Informe um valor maior que zero."`. Esse validador **intercepta primeiro** (validação no submit do form). Task 4 + Task 3 (mensagem `"Valor da custa deve ser maior que zero"`) NUNCA disparam.

**Resultado:** Defesa em profundidade tripla **funcional** (zero submit passa com valor=0), mas mensagem **inconsistente** entre 3 camadas:
1. Client preexistente (Form): "Informe um valor maior que zero."  ← dispara primeiro
2. Client Task 4 (handler): "Valor da custa deve ser maior que zero" ← nunca dispara
3. Server Task 3 (service): "Valor da custa deve ser maior que zero" ← nunca dispara

**Lição:** Phases que adicionam validação anti-X (anti-zero, anti-null, anti-duplicate) devem grep client validadores preexistentes ANTES de assumir que novo guard será exercitado. Pattern típico:
```bash
grep -rn "Informe um\|Valor.*zero\|deve ser maior" src/components/Form*.jsx
```

Decidir explicitamente:
- **Opção A**: deixar preexistente, novo guard só como defesa em profundidade (e marcar string drift como deferred)
- **Opção B**: remover validador preexistente, novo guard vira primário
- **Opção C**: unificar string entre as 3 camadas pré-execute

**Memory feedback NOVO:** `preexisting_form_validators_must_be_audited_in_anti_X_phases`

**Aplicação:** Em phases anti-X de validação, Task 1 deve incluir audit de "validators existentes que cobrem a mesma propriedade". Sem audit, phase entrega defesa em profundidade não-planejada (positivo) com mensagens divergentes (negativo, defer pra phase de unificação).

### Lição 4 — UAT humano em prod pega bugs que gates automatizados não pegam (REFORÇO de cumulative pattern)

**Sintoma cumulative:** 3 phases recentes detectaram bugs críticos APENAS em UAT humano em prod, todos passando build + test:regressao + lint + Gate D-01:
- **Phase 7.10.bug3 SC3** — bug dbDelete signature (sintaxe válida, semântica errada PostgREST)
- **Phase 7.13d SC4b** — bug "input N → DB N-x" não-determinístico em DetalheContrato edit
- **Phase 7.13b UAT** — 4 bugs (3 fixed via Migrations 005+006 + D-pre-15 + 1 deferred 7.13c)

**Lição:** Gates automatizados (build, test, lint, Gate D-01 dual) cobrem:
- Sintaxe correta
- Tipos válidos (TypeScript se houver)
- Tests não regridem
- Motor crítico não foi tocado
- Imports resolvem

Gates automatizados **NÃO cobrem**:
- Semântica de chamadas REST (PostgREST URL bem-formada)
- Comportamento real do banco (FK CASCADE funciona, RLS permite, etc)
- Pipeline cliente→servidor→DB lossless (parsers, encoders, etc)
- UI states que dependem de side-effects (useEffect, hooks paralelos)
- Drift entre validadores em camadas

**Memory feedback REFORÇO:** `uat_humano_pega_bugs_que_gates_automatizados_nao_pegam` (cumulative pattern, 3 phases consecutive).

**Aplicação:** UAT humano em prod NÃO é checkpoint opcional — é o **detector primário** pra bugs de semantic-drift, side-effect, e pipeline integration. Phases que envolvem (a) helpers compartilhados, (b) novos validadores, (c) refactors estruturais devem ter UAT humano explícito ANTES de tag/SHIP.

### Lição 5 — Defesa em profundidade tripla é robusta mesmo sem unificação (NOVA)

**Sintoma SC4:** anti-zero protegido por 3 validadores em camadas com mensagens divergentes. Plan original assumia 2 (Task 3 + Task 4), preexistente foi descoberto durante UAT.

**Lado positivo (não trivial):** mesmo se Task 3 OU Task 4 quebrar no futuro (ex: refactor que remove validation accidentally), o validador preexistente **continua bloqueando** submits com valor=0. Robustez ≠ unificação — pode ser trade-off aceitável quando descoberta é tardia.

**Lição:** Robustez é **inverso** de unificação:
- **Unificação** = 1 source of truth, mais fácil de manter, ZERO drift
- **Robustez** = N sources of truth, drift possível, MAIS difícil de quebrar acidentalmente

Phase aceita defesa em profundidade tripla com nota de "string drift defer pra phase de unificação" quando:
- Funcionalidade está garantida (zero falha funcional)
- Drift é cosmético (operador entende ambas mensagens)
- Phase de unificação tem severidade BAIXA (zero perda receita ou bug crítico)

**Aplicação:** Quando descoberta tardia traz "validador adicional não-planejado", documentar trade-off explicitamente em SUMMARY/LEARNINGS — não tratar como bug, é robustez de bônus. Phase de unificação vira deferred, não emergencial.

## Patterns observados (3)

### Pattern 1 — ABORT GATE como detector de scope drift (vs scope creep)
ABORT GATE pré-DELETE foi desenhado pra detectar divergência **numérica** (count != esperado). Funcionou pra detectar divergência **categórica** (3 advair NÃO eram órfãs vazias, eram qtd=1). Pattern: gates numéricos podem detectar drift categórico se a query for granular o suficiente (Q14 com FILTER por categoria + Q B preview com SELECT *).

### Pattern 2 — Fix-forward 1-linha pós-UAT vs ABORT total
Fix-forward funcionou porque:
- Bug era **isolado** (1 linha, sem dependências cumulative)
- Padrão correto JÁ EXISTIA em prod (L393 + L471)
- D-01 motor INTOCADO preservado durante fix
- Vercel deploy auto-disparado

Quando NÃO usar fix-forward: bug requer refactor amplo, dependencies cumulative, ou D-01 motor é tocado.

### Pattern 3 — Scope reduction in-place via PLAN.md edit
PLAN.md commitado (`c64bc98`) foi editado in-place no commit `352366f` pós-ABORT GATE. Funciona porque:
- PLAN.md é audit trail vivo, não imutável
- Edits documentam scope reduction com rastreabilidade (mensagem commit explícita)
- ROADMAP.md captura a phase nova BACKLOG (Phase 7.10.bug5)

Pattern complementar a "PLAN.md aprovado é imutável" — aprovado = state inicial, mid-execução discoveries podem editar com commit explícito.

## Surprises (4)

### Surprise 1 — Conferência visual de 7 órfãs ANTES de SQL (sanity humano)
Operador conferiu visualmente 7 linhas R$ 0,00 / R$ 0,00 no Módulo Dívidas filtro "mende" (paginação 4+3) ANTES de rodar Query 7. Métrica humana confirmou estabilidade entre investigação 30-abr e execução 30-abr. Pattern saudável: sanity humano ≠ redundante com SQL gates, é confirmação cross-modal.

### Surprise 2 — Counter UI "139 em cobrança" diferente de `total_dividas=143`
Pós-DELETE: `total_dividas=143` (DB). Counter UI mostrou "139 em cobrança". Diferença: counter UI filtra por status "Em Cobrança" (exclui Quitadas, Arquivadas, etc). Counter UI é **agregação semântica diferente** do COUNT(*) bruto. Lição: comparar métricas DB vs UI sempre considerar filtros UI implícitos.

### Surprise 3 — 26 linhas filtro "mende" (não 13)
PLAN esperava 13 linhas (estimativa CONTEXT.md). Realidade: 26 linhas pq filtro "mende" pega:
- 13 custas reais MENDES (Valor Original > 0)
- 12 parcelas do acordo MENDES (R$ 1.000 cada)
- 1 linha extra (paginação ou outro)

Lição: estimativas numéricas em CONTEXT.md baseadas em snapshot SQL podem divergir de realidade UI por causa de joins/relacionamentos não-óbvios. Métrica crítica deve ser **invariante semântica** (zero R$ 0,00 simultâneo), não count exato.

### Surprise 4 — Bug semântico passou TODOS os gates automatizados
Build + test:regressao 34/34 + Gate D-01 dual + lint (não configurado mas hipotético) + Vercel deploy + console limpo no carregamento. **Bug só foi pego no clique humano "Excluir"**. Lição cumulative: gates automatizados são NECESSÁRIOS mas NÃO SUFICIENTES — UAT humano é detector primário pra semantic-drift de helpers compartilhados.
