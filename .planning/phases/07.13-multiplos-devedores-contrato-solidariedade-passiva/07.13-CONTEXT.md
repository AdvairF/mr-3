<domain>
## Phase Boundary

Suporte a **N devedores por contrato** com solidariedade passiva (CC art. 264-285 e 818-839). Hoje `contratos_dividas.devedor_id` é single-FK BIGINT NOT NULL. Esta phase introduz a capacidade do advogado cadastrar **múltiplos devedores** (principal + coobrigados/fiadores/avalistas) num mesmo contrato, refletido em UI cadastro (DetalheContrato), listagens (ModuloContratos, Pessoas) e detalhes (DetalheDivida).

**Implementação reusa junction `devedores_dividas` existente** (NÍVEL DÍVIDA, 6 papéis, coluna `responsabilidade`) — sem schema migration nova. Vinculação no contrato = fan-out para N rows na junction (uma por dívida do contrato).

Fora de escopo (deferred):
- Auto-povoamento de polo passivo em ProcessosJudiciais a partir dos devedores do contrato
- Ativação semântica de `responsabilidade=SUBSIDIARIA` (benefício de ordem CC art. 827)
- Mudança de papéis em UI Pessoas (cross-contract)
- Decomposição/petição multi-devedor

</domain>

- **D-pre-5 (Schema/arq, REVISADA — N/A):** Migração retrocompat dispensada — junction já populada via `seedPrincipal` (App.jsx L3264 + service `devedoresDividas.js:48-69`). Ambos contratos atuais (MENDES + advair) já têm 1 row PRINCIPAL por dívida na junction (verificado).

- **D-pre-6 (UX, MANTIDA):** Fiador (e qualquer não-PRINCIPAL) aparece em **Pessoas com saldo cheio** do contrato (R$ 36.652,56 no exemplo MENDES). Mudança intencional de UX: listagem Pessoas vira "perspectiva de cobrança individual" (cada devedor = quanto pode ser cobrado dele = todo, por solidariedade passiva). Carteira Total Dashboard **preserva dedupe atual** via `papel=PRINCIPAL` filter (App.jsx L597-608) — zero mudança Dashboard.

- **D-pre-7 (Schema/arq, SUBSTITUÍDA):** **Manter os 6 papéis existentes** — PRINCIPAL, COOBRIGADO, AVALISTA, FIADOR, CONJUGE, OUTRO. UI 7.13 expõe TODOS os 6 (consistente com `DevedoresDaDivida.jsx` PAPEL_META L5-12). Distinção é semântica/jurídica; cálculo é idêntico para todos não-PRINCIPAL (D-pre-8 abaixo).

- **D-pre-8 (Schema/arq, SUBSTITUÍDA):** Coluna `responsabilidade` **MANTIDA** no schema (SOLIDARIA, SUBSIDIARIA, DIVISIVEL). UI 7.13 default = SOLIDARIA, **não expõe seletor** (UX limpa, schema pronto). Cálculo de saldo/agregação ignora `responsabilidade` em 7.13 — todos os papéis tratados como SOLIDARIA. Phase futura ativa SUBSIDIARIA (benefício de ordem) sem precisar migration.

- **D-pre-9 (UX, NOVA):** UI cadastro **SÓ POR CONTRATO**. DetalheContrato.jsx ganha componente multi-devedor (estilo `DevedoresDaDivida.jsx` mas escopo contrato). DetalheDivida exibe lista de devedores **read-only** — sem botões adicionar/remover/promover/✕. Edição cascateia do contrato; dívida individual não permite override (menor superfície de inconsistência).

- **D-pre-10 (Cascade, NOVA):** Vincular devedor X com papel Y ao contrato C → **N inserts** em `devedores_dividas` (uma row por dívida do contrato C), todas com mesmo (devedor_id=X, papel=Y, responsabilidade='SOLIDARIA'). Remover devedor X do contrato C → **N deletes**. Alterar papel (sem mudar PRINCIPAL) → **N PATCHes**. Operações atômicas no nível do app (não trigger DB) — failure mid-loop deixa estado parcial reexecutável (mesmo trade-off de `excluirContrato` Phase 7.2 D-04). Promoção a PRINCIPAL com PRINCIPAL anterior existente NÃO é trivial PATCH — segue wizard D-pre-13.

- **D-pre-11 (Cascade, NOVA):** Adição de **nova dívida** em contrato com múltiplos devedores → cascade existente `seedPrincipal` é **estendida** para popular junction com TODOS os devedores do contrato (não só o `contrato.devedor_id` legado). Novo helper: `seedDevedoresDoContrato(dividaId, contratoId)` em `services/devedoresDividas.js` ou `services/contratos.js`. Chamado de `adicionarDocumento` (após cada `dbInsert("dividas", ...)`) e de criação manual de dívida (App.jsx fluxo legacy se ainda houver).

- **D-pre-12 (Validações, NOVA):** App-level enforcement:
  - **≥ 1 PRINCIPAL** ao salvar contrato (não trigger DB; primeiro devedor adicionado vira PRINCIPAL automático via `seedPrincipal` atual).
  - **Bloquear remoção do último PRINCIPAL** com warning (Modal.jsx, sem `window.confirm` — D-05 LOCKED em 7.10/precedentes). Usuário precisa promover outro a PRINCIPAL antes de remover o atual.
  - **UNIQUE INDEX `papel=PRINCIPAL` por dívida** (existente, schema 002 L172) — **preservado**, garante ≤ 1 PRINCIPAL.
  - Promoção a PRINCIPAL com PRINCIPAL anterior existente segue wizard D-pre-13 abaixo (sobrepõe `demoverPrincipalAtual` automático silencioso de `devedoresDividas.js:71-83`).

- **D-pre-13 (Validações, NOVA):** **TROCA DE PAPEL VIA WIZARD 2-STEPS COM CONFIRMAÇÃO.** Promover devedor não-PRINCIPAL para PRINCIPAL quando contrato JÁ TEM PRINCIPAL:
  1. Modal de confirmação obrigatório: `"Contrato já tem PRINCIPAL ([nome atual]). Promover [nome novo] a PRINCIPAL e demover [nome atual] a:"`
  2. Dropdown com 5 papéis não-PRINCIPAL: COOBRIGADO | SOLIDARIO | FIADOR | AVALISTA | CONJUGE | OUTRO. **Sem default automático.** Usuário escolhe.
  3. Confirma → N PATCHes em `devedores_dividas` (N = nº dívidas do contrato): N rows com `devedor_id=novo` recebem `papel='PRINCIPAL'`; N rows com `devedor_id=anterior` recebem o papel escolhido pelo usuário.
  4. UNIQUE INDEX (`papel='PRINCIPAL'`, `divida_id`) preservado: 1 PRINCIPAL por dívida em todo momento.
  5. Validação D-pre-12 mantida: ≥1 PRINCIPAL preservado (novo assume).

  **NÃO permitir promoção direta sem confirmação.** Sobrepõe `demoverPrincipalAtual` silencioso (devedoresDividas.js:71-83) — wizard explícito, sem ação automática. Justificativa: "auto-demovido a COOBRIGADO" da proposta inicial era invenção; usuário escolhe explicitamente o novo papel do PRINCIPAL anterior, evita assumir intent.

- **D-pre-14 (UX, NOVA):** ModuloContratos listagem inline — query expandida retorna devedores do contrato distinct (via JOIN `devedores_dividas` → `dividas` → `contratos_dividas`, agrupando por contrato). Render `"Nome1 (Papel1), Nome2 (Papel2)"`. Devedor PRINCIPAL aparece primeiro (sort por papel: PRINCIPAL → outros).
