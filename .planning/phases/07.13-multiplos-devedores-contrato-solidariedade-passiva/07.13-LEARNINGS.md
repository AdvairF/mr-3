## Patterns

### Junction reuse com fan-out N rows
Para "set de X aplicado a um agregador" (ex: devedores aplicado a contrato), preferir reusar junction nível-mais-baixo (dívida) com fan-out por agregador (contrato) ao invés de criar nova tabela nível-agregador. Cascade: vincular X ao contrato C = N inserts (1 por dívida do contrato).

**When to use:** quando junction nível-baixo existe e tem infraestrutura (UI, hooks, queries existentes). Evita 2 fontes de verdade conflitantes.
**Source:** 07.13-CONTEXT.md (D-pre-1, D-pre-4)

---

### Cascade fan-out N inserts/deletes/PATCHes — operações atômicas no app
N operações sequenciais ao invés de transação DB. Failure mid-loop deixa estado parcial reexecutável (mesmo trade-off de `excluirContrato` Phase 7.2 D-04).

**When to use:** quando schema não permite transações ergonômicas (Supabase REST PostgREST sem support de tx ergonômico) E operação é idempotente em retry. Trade-off: simplicidade > atomicidade DB.
**Source:** 07.13-01-PLAN.md (D-pre-10)

---

### Wizard 2-steps com confirmação obrigatória
Para mudanças destrutivas com múltiplos resultados possíveis (ex: promover X a PRINCIPAL exige decidir o que fazer com PRINCIPAL anterior), wizard com dropdown obrigatório SEM default + botão Confirmar disabled até escolha.

**When to use:** quando ação tem múltiplas opções e nenhuma é "default seguro". Evita ação silenciosa que usuário não escolheu explicitamente.
**Source:** 07.13-CONTEXT.md (D-pre-13) + 07.13-01-PLAN.md (Task 1 helper `promoverParaPrincipalComDemocao`)

---

### Modal warning sem `window.confirm` (D-05 LOCKED)
Bloqueio absoluto sem opção de bypass via Modal customizado. Texto explicativo + único botão "Entendido".

**When to use:** quando schema constraint (UNIQUE INDEX, NOT NULL, FK) impede ação E usuário precisa entender por quê. `window.confirm` não comunica bem; Modal customizado dá texto educativo.
**Source:** 07.13-01-PLAN.md (Task 3 BloqueioRemocaoUltimoPrincipalModal)

---

### Helper paralelo > modificar original
Criar `XIncluindoSolidarios` ao invés de modificar `X`. Original preservado para callsites que dependem do behavior antigo.

**When to use:** quando 2 callsites têm semânticas opostas (Dashboard precisa dedupe vs Pessoas precisa fan-out). DRY relaxed: 80% código duplicado vs 100% break em um lado.
**Source:** 07.13-01-SUMMARY.md (helper paralelo Task 6 + helper paralelo fix-forward)

---

### Read-only mode via prop default false
Componente compartilhado ganha `readOnly = false` default. Callsites que querem read-only passam `readOnly={true}`. Internal conditionals `{!readOnly && ...}` escondem botões.

**When to use:** quando UI multi-nível precisa do mesmo componente em modo edit (nível superior) E read-only (nível inferior). Default false preserva back-compat.
**Source:** 07.13-01-PLAN.md (Task 4 DevedoresDaDivida prop readOnly)

---

### Final Gates Task sem commit dedicado
Última task do plan = verificação cumulativa programática (build + tests + invariantes + shields) sem `git commit` dedicado. Substitui antigo `chore(*): shields` commit. SUMMARY untracked até commit docs consolidado.

**When to use:** quando plan tem 6+ commits feat e shields/gates não introduzem mudança em produção (verificação only).
**Source:** 07.13-01-PLAN.md (Task 7)

---

### Dividas-via-junction injection — view-devedor `{...d, dividas: fanout}`
Construir devedor "view" com `dividas` substituídas pelo fan-out via junction antes de passar para motor. Motor permanece intocado (D-01 strict).

**When to use:** quando motor lê campo legacy single-FK que precisa ser fan-out via junction sem modificar motor. Pattern análogo já existia em Dashboard L597-608 para FILTRAR (anti-dupla-contagem); aqui aplicado para EXPANDIR (Pessoas saldo cheio).
**Source:** 07.13-01-SUMMARY.md (fix(07.13f.bug) helper inline `devedorWithDividasViaJunction`)
