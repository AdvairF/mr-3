## 1. Problema

`calcularSaldosPorDivida` em `devedorCalc.js:172-176` filtra `_so_custas:true` ANTES do cálculo:

```js

  (d) => !d._nominal && !d._so_custas

```

Consequência: `saldosMap[id_so_custas] === undefined` para toda dívida-fantasma de custas avulsas (criadas em Phase 7.9 via `services/contratos.js` D-23).

UI consumers iteram TODAS as dívidas mas não tratam o subset filtrado:

- **`TabelaDividas.jsx:94-96`** — `saldo = saldosMap[String(d.id)] ?? null`; `saldoExibido = d.saldo_quitado === true ? 0 : saldo`.
- **`TabelaDividas.jsx:134`** — `saldoExibido == null ? "Calculando..." : fmtBRL(saldoExibido)`.
- **`DetalheDivida.jsx:79-86`** — pattern equivalente; `saldoAtual` cai pra `null` → mesma label "Calculando..." (não-fix neste plan se for outro componente; mas usuário citou L79-86 — confirmado no HEAD).

Pattern de label "Calculando..." é leak de `undefined` síncrono mascarado de loading state (memória 2026-04-25 ponto 4).

**Frequência confirmada via UAT prod 2026-05-01**: ~50% das linhas do módulo Dívidas mostram "Calculando..." indefinido; todas as afetadas têm Valor Original R$ 0,00 (assinatura clássica de `_so_custas:true`, criadas pelo fluxo D-23 da 7.9 que sempre cria dívida-fantasma com `valor_total: 0`).

## 5. D-pre lockeds

### D-pre-1 — Helper único `calcularValorAtualizadoCustasAvulsas(custas, hoje)`
Arquivo NOVO `src/utils/custasAvulsas.js`. Assinatura:
```js
export function calcularValorAtualizadoCustasAvulsas(custas, hoje) {
  if (!Array.isArray(custas) || custas.length === 0) return 0;
  let total = 0;
  for (const c of custas) {
    const vc = parseFloat(c.valor) || 0;
    if (c.data && c.data < hoje) {
      total += vc * calcularFatorCorrecao("inpc", c.data, hoje);
    } else {
      total += vc;
    }
  }
  return total;
}
```
Importa `calcularFatorCorrecao` de `./correcao.js` (mesma fonte que motor L485 usa).

### D-pre-2 — `TabelaDividas.jsx:94-96` ganha branch `_so_custas`

const saldoCustas = d._so_custas

### D-pre-5 — Helper NÃO filtra `c.pago` (resolvido pelo precedente)
Espelhamento exato de `devedorCalc.js:479-491` — motor itera `for (const c of div.custas || [])` SEM filtro por `c.pago`.

**Razões locked:**
1. **Precedente Phase 7.9 P3** (`DetalheContrato.jsx:637-641`): "correção sempre aplicada (independente de c.pago). Advogado paga a custa de antemão e cobra correção desde a data do pagamento."
2. **Memory `feedback_custas_semantica_quitacao_dual`**: `c.pago` em custas é dual-semântica — significa "advogado pagou ao judiciário", NÃO "devedor quitou a dívida". Filtrar `c.pago === true` zerar a custa avulsa seria erro semântico (advogado pediu ressarcimento ao devedor, com correção INPC desde a data do pagamento).
3. **Single source of truth**: helper que diverge do motor reintroduz a classe de bug que D-01 protege.

Lição transferível: nunca tratar `c.pago` em isolamento — sempre delegar à fórmula do motor.

### D-pre-6 — `data` shape `c.data` (não `c.data_pagamento`)
Motor L484 usa `c.data` (legacy shape D-22). Helper segue mesmo campo. Não normalizar para `c.data_pagamento` (campo separado, não usado pelo motor pra cálculo de fator).

### D-pre-7 — Test parity: helper === motor para mesmo input
Test obrigatório que feed mesma `custas[]` no motor (`calcularDetalhesAtualizados` agregando `totalCustasAtualizado` da seção `_so_custas`) e no helper, e assert igualdade ±1e-9. Gate de drift.

## 6. Tests planejados

`src/utils/__tests__/custasAvulsas.test.js` (NOVO):
- Custa única sem `data` → soma valor puro.
- Custa única com `data` < hoje → soma `vc * calcularFatorCorrecao("inpc", c.data, hoje)`.
- Custas múltiplas (mix com/sem data) → soma cumulativa.
- Custa com `valor` inválido (string vazia, null, undefined) → ignora (`parseFloat || 0`).
- Custas com `c.pago === true` → contribuem normal (D-pre-5: NÃO filtra).
- Array vazio ou `null` → retorna 0.
- **Parity test (D-pre-7)**: motor `calcularDetalhesAtualizados` vs helper, mesmo fixture, igualdade ±1e-9.

Regressão `devedorCalc.test.js` (34 testes existentes) — PASS sem modificação.
