# Phase 7.10bcd Learnings: Migração 3 consumidores (Devedores + ModuloDividas + Dashboard) para pagamentos_divida

## Decisions

### D-30 — 3 commits atômicos no submódulo (NÃO 1 commit gigante)
Migrar 3 consumidores em chain `feat(07.10b): Devedores...` → `feat(07.10c): ModuloDividas...` → `feat(07.10d): Dashboard...` em vez de 1 commit `feat(07.10bcd)` consolidado.

**Rationale:** Bisect funciona; isolamento de regressão por consumer; sub-PAUSA humana após cada commit valida regressão antes de prosseguir (impede acumular bug serial). Risco fast-track (3x mais surface) mitigado por granularidade.
**Source:** 07.10bcd-CONTEXT.md §4 D-30, 07.10bcd-01-PLAN.md, 07.10bcd-01-SUMMARY.md

### D-31 — Helper compartilhado `agruparPagamentosPorDevedor`
Criar helper novo (`src/utils/agruparPagamentosPorDevedor.js` + test 2 it()) no commit 07.10b (1º consumidor que precisa) e reutilizar em commits 07.10d (Dashboard L568 + parent App L8425).

**Rationale:** DRY > inline duplication. 2 callsites principais + complexidade da lookup map (intermediário `divida_id → devedor_id` por causa de schema sem `devedor_id` direto em `pagamentos_divida`) justificam extração; lição "helper-first" da 7.9 aplicada antecipadamente. Test trivial blinda regressão silenciosa do helper (lição shield wrapper equivalência 7.8.2a).
**Source:** 07.10bcd-CONTEXT.md §4 D-31, 07.10bcd-01-SUMMARY.md

**Rationale:** Defesa contra empilhamento de regressão silenciosa em fast-track. Granularidade humana > granularidade automatizada quando bug pode acumular cross-commit.

## Lessons

### Bug pré-existente expõe-se durante UAT por dataset criado em phase recente
Custas avulsas (Phase 7.9 feature) criadas durante UAT 7.9 expuseram gap UI (`saldosMap[id_so_custas] === undefined` → "Calculando..." indefinido) que existia desde 7.9 mas não havia sido trigado anteriormente.

**Context:** Hipótese inicial do usuário ("Set lookup novo da 7.10c retorna `[]` pra `_so_custas`") era **tecnicamente correta mas IRRELEVANTE**: motor `calcularSaldosPorDivida` já filtrava `_so_custas` ANTES do cálculo desde 7.9, independente da fonte de pagamentos. Pre-7.10bcd e pos-7.10bcd têm comportamento idêntico para essas linhas. Cross-check em PROD 7.10a confirmou comportamento igual.
**Source:** Conversa UAT PAUSA #1 (absorvida em 07.10bcd-01-SUMMARY.md §"Bug pré-existente _so_custas"), `memory/feedback_so_custas_ui_calculando_indefinido.md`

### Fast-track UAT requer pre-check de dataset (regulares vs degenerados)
Antes de invocar D-34 rollback, verificar se sintoma reproduz em PROD com mesma versão pré-migração. Phases consumer-side migration com motor `D-01 strict` raramente introduzem bugs visuais de subset filtrado pelo motor — quase sempre são pré-existentes.

**Context:** Lição registrada em `memory/feedback_so_custas_ui_calculando_indefinido.md` aplicada DURANTE a 7.10bcd UAT — economizou rollback total desnecessário (que NÃO resolveria o bug pré-existente).
**Source:** UAT PAUSA #1 diagnóstico + `memory/feedback_so_custas_ui_calculando_indefinido.md`

### Drift D-28 pode ser CORREÇÃO esperada, não regressão
Drift do `advair` (PROD R$ 939,90 vs LOCAL R$ 541,29 = -R$ 398,61) parecia regressão à primeira vista. Análise mostrou: PROD valor inflado (não descontava 3 pagamentos via stored proc); LOCAL valor com desconto (canônico). Phase 7.10bcd corrigiu bug pré-existente de consumers legacy (Devedores/Dashboard liam pagamentos_parciais que nunca recebeu esses pagamentos).

**Context:** D-28 originalmente documentado como "drift de pagamentos novos via UI legacy órfãos pós-migração". Aqui foi o **inverso**: pagamentos via stored proc Phase 7 que não eram visíveis em consumers legacy passaram a ser visíveis em LOCAL pós-7.10bcd. Sintoma idêntico ao D-28 mas semântica oposta (correção, não trade-off).
**Source:** UAT SC2/SC5/SC6/SC7 análise (07.10bcd-01-SUMMARY.md §"UAT Plan 02 PAUSA #1")

### Validação cross-check de coerência detecta bugs sutis melhor que valor isolado
Δadvair em SC2b (-R$ 398,61) ≡ ΔCARTEIRA em SC5a/b (-R$ 398,61) ≡ Header CARTEIRA = Dashboard CARTEIRA TOTAL = parent App L8425 Map. Coerência ponta-a-ponta valida Edit 5 D-extra e confirma helper compartilhado funciona em todas as callsites.

**Context:** Drifts isolados parecem aleatórios; coerência transitiva entre entidade-individual e agregado-total comprova que mesmo helper produz mesmo resultado. Detectou que Header global e Dashboard interno usam mesma fonte (sem regressão silenciosa entre os 2 Maps `pgtosPorDevedorCarteira`).
**Source:** UAT SC5a/b/c análise (07.10bcd-01-SUMMARY.md §"Validação cruzada de coerência")

### Gates automatizados (build + regressão + grep) NÃO cobrem dataset edge cases
Build OK + Regressão 25/25 + 4 Shields OK + D-01 strict não detectaram o bug pré-existente `_so_custas` — porque tests não cobrem dívidas-fantasma com `valor_original=0`. Apenas UAT humano com dataset real (custas criadas durante 7.9 UAT) trigou o sintoma.

**Context:** Lição complementar à `feedback_db_integration_gate_missing.md` (7.9): gates Plan 01 (build + unit + grep) não pegam INSERT real, FK NOT NULL, helper duplication, semantic drift entre callsites — agora também não pegam UI gap em entidades degeneradas filtradas pelo motor.
**Source:** UAT PAUSA #1 diagnóstico, `memory/feedback_db_integration_gate_missing.md`

### Hooks PreToolUse READ-BEFORE-EDIT podem fire retroativamente sem bloquear edits
Edits aplicados com sucesso ("file state is current in your context") apesar de hook reminders pedirem Read prévia. Reminders são informacionais quando harness já tracks file state — não invalidam edits bem-sucedidos.

**Context:** Confusão potencial: assumir que hook = falha. Verificar success message do tool result antes de re-Read desnecessário.
**Source:** Conversa execução Tasks 3-4 (múltiplos edits em App.jsx)

## Patterns

### D-27 Set lookup `divida_id ∈ dividas-do-devedor` (replicado 4x)
Pattern reutilizável quando schema novo (`pagamentos_divida`) não tem `devedor_id` direto:
```js
const dividaIdsDoDevedor = new Set((devedor.dividas || []).map(d => String(d.id)));
const pagamentosDoDevedor = pagamentosDivida.filter(p => dividaIdsDoDevedor.has(String(p.divida_id)));
```

**When to use:** Migrações consumer-side de fonte com `devedor_id` direto → fonte sem (apenas FK `divida_id`). Inline OK até 5 callsites; ≥5 vira helper. Aplicado em TabelaDividas L41-46 + DetalheDivida L73-76 + Dashboard ultimosPagamentos L693-708 + Dashboard recuperadoPeriodo L680-692.
**Source:** 07.10bcd-01-SUMMARY.md §"D-27 Set lookup pattern reutilizável", herdado da 7.10a

### D-31 Helper compartilhado para agrupamento por devedor
`agruparPagamentosPorDevedor(devedores, pagamentosDivida) → Map<devedorId, Pagamento[]>` com lookup intermediário `dividaIdToDevedorId` interno.

**When to use:** Quando 2+ callsites precisam Map agrupador idêntico E lookup interno tem complexidade (não trivial 1-liner). 3 callsites em App.jsx (L571 Dashboard interno + L2994 Devedores + L8430 parent App) justificaram extração. Test trivial 2 it() blinda regressão silenciosa.
**Source:** 07.10bcd-01-SUMMARY.md §"D-31 helper compartilhado", `memory/feedback_db_integration_gate_missing.md` (lição helper-first)

### D-30 commits atômicos sequenciais bisect-able (1 por consumer)
Em phases multi-consumer, commit por consumer com header `feat(NN.NN[letra]): <consumer>...` em chain sequencial. Cada commit deve passar regressão + gates D-01 + Shield específico antes do próximo.

**When to use:** Phases que tocam ≥3 consumers com escopo similar. Trade-off: mais ceremony (3 commit messages) vs git bisect funciona + sub-PAUSAs humanas validam por commit. Headers exatos verificáveis via `git log baseline..HEAD --format="%s" | grep -cE "^feat\(NN\.NN[letra]\):"` deve retornar count exato.
**Source:** 07.10bcd-CONTEXT.md §4 D-30, 07.10bcd-01-PLAN.md gates D-30

### Dual gate D-01 strict (C1 diff + C2 --name-only)
Verificar motor 100% intocado em phase consumer-side via 2 gates complementares:
- **C1**: `git diff baseline..HEAD -- <motor files>` → 0 lines (working tree + commits)
- **C2**: `git log baseline..HEAD --format="" --name-only | sort -u | grep -E "<motor files>"` → 0 matches

**When to use:** Sempre que phase deve preservar D-01 strict cumulativo. C1 sozinho não pega cenário de motor commitado e revertido em commit posterior; C2 sozinho não pega working tree com edits não commitados. Use `--name-only` (NÃO `--stat`) para evitar falsos positivos por match em commit message (lição `memory/feedback_gate_motor_grep_use_name_only.md` 7.8.2a 608c124).
**Source:** 07.10bcd-01-PLAN.md gates C1/C2, `memory/feedback_gate_motor_grep_use_name_only.md`

### Cross-check de coerência ponta-a-ponta para validar agregação
Validar que delta de uma entidade individual (advair -R$ 398,61) ≡ delta do agregado (CARTEIRA -R$ 398,61) ≡ delta de cada callsite paralelo (Header CARTEIRA = Dashboard CARTEIRA TOTAL). Coerência transitiva valida que helper compartilhado produz mesmo resultado em N callsites.

**When to use:** UAT de phases que migram fonte com múltiplos consumers/agregados que devem ler mesma fonte canônica. Drifts isolados parecem aleatórios; coerência transitiva é diagnóstico forte de regressão entre callsites.
**Source:** UAT PAUSA #1 análise SC2/SC5/SC6/SC7

### Grep global pre-execute para descobrir callsites não-óbvias
Antes de execute, grep global por padrão alvo (`allPagamentos.forEach`) em todo o arquivo (não apenas regiões mencionadas em CONTEXT/PLAN). Lição `feedback_grep_global_before_pointwise_fix.md` (7.9 rounds 2-4).

**When to use:** Phases consumer-side que tocam ≥2 regions do mesmo arquivo. Edit 5 D-extra (parent App L8425) só foi descoberto pre-execute via grep global — sem ele, header card "Carteira" continuaria divergente. Custo: ~30s grep; ganho: previne hot-fix pós-ship.
**Source:** 07.10bcd-CONTEXT.md §2 Fix D-extra, `memory/feedback_grep_global_before_pointwise_fix.md`

## Surprises

### 4ª callsite Map agrupador descoberta pré-execute (Edit 5 D-extra)
CONTEXT original previa 3 consumers (Devedores + ModuloDividas + Dashboard). Grep global pre-execute por `allPagamentos.forEach` revelou **4ª callsite no parent App L8425** (`pgtosPorDevedorCarteira` global que alimenta header "Carteira" visível em TODAS as tabs).

**Impact:** Sem Edit 5 D-extra, header card "Carteira" do header global continuaria divergente do Dashboard interno pós-ship. Custo: +1 edit (5 total em App.jsx no commit 07.10d). Ganho: cobertura completa, sem hot-fix pós-ship. Lição `feedback_grep_global_before_pointwise_fix.md` aplicada antecipadamente economizou esse hot-fix.
**Source:** 07.10bcd-CONTEXT.md §2 Fix D-extra, 07.10bcd-01-SUMMARY.md

### Bug pré-existente exposto por dataset criado em phase recente (não em PROD legacy)
Custas avulsas (`_so_custas:true`) introduzidas em Phase 7.9 e criadas pelo usuário durante UAT 7.9 expuseram gap UI ("Calculando..." indefinido) em phase 7.10bcd UAT — feature recente expondo bug recente que escapou ao seu próprio UAT.

**Impact:** Quase disparou D-34 rollback total (que NÃO teria resolvido). Diagnóstico forçou cross-check em PROD 7.10a (mesmo comportamento confirmado) → bug categorizado como pré-existente → Phase 7.10.bug2 backlog. Tempo extra: ~30min UAT investigação. Lição `feedback_so_custas_ui_calculando_indefinido.md` registrada para futuras phases.
**Source:** UAT PAUSA #1 conversa, `memory/feedback_so_custas_ui_calculando_indefinido.md`

### Drift D-28 do `advair` foi CORREÇÃO esperada, não regressão (semântica oposta ao planejado)
D-28 documentava drift de "pagamentos via UI legacy órfãos pós-migração" (esperado: pagamentos novos não aparecem em consumers migrados). Realidade UAT: drift foi de "pagamentos via stored proc Phase 7 que não eram visíveis em consumers legacy" (LOCAL canônico, PROD inflado).

**Impact:** Confirmação positiva inesperada — phase 7.10bcd não apenas migrou consumers como **corrigiu bug pré-existente** de consumers atrasados que não viam pagamentos via stored proc. Smoke prod pos-ship: prod alinhou com LOCAL (advair R$ 541,29). Validou que phase 7.10bcd tinha valor de **bug fix** além de tech debt cleanup.
**Source:** UAT SC2 análise + smoke prod confirmação (07.10bcd-01-SUMMARY.md)

### Build OK + 25/25 regressão + 4 Shields OK não detectaram bug pré-existente
Todos os gates Plan 01 verdes (Build + Regressão 25/25 + Gate C1/C2 + Shields 24a/b/c/d 15 sub-checks) **não pegaram** o bug `_so_custas` "Calculando..." indefinido. Apenas UAT humano com dataset real trigou.

**Impact:** Reforça lição `feedback_db_integration_gate_missing.md` (7.9): gates automatizados não cobrem (a) DB integration real, (b) semantic drift entre callsites, (c) **dataset edge cases** (entidades degeneradas filtradas pelo motor — adicionado nesta phase). UAT humano continua sendo guardrail final crítico mesmo com 12 gates automatizados verdes.
**Source:** UAT PAUSA #1 diagnóstico, comparação com gates Plan 01 Task 5 matriz consolidada

### Phase fast-track concluiu com ZERO retrabalho semântico
Apesar do risco MÉDIO-ALTO (3x mais surface vs 7.10a), 6 mitigations M1-M6 aplicadas + lições anteriores (helper-first 7.9 + grep global 7.9 + dual gate motor 7.8.2a + shield wrapper equivalência 7.8.2a) blindaram a phase. **Zero rollback, zero hot-fix dentro da phase, zero regressão pos-ship**. Bug `_so_custas` foi pre-existing (não imputável a 7.10bcd).

**Impact:** Validou estratégia fast-track como viável em phases consumer-side com motor D-01 strict + lições prévias acumuladas. Custo: 5 humanos checkpoints (3 sub-PAUSAs intra-Plan-01 + 2 PAUSAs Plan 02). Tempo total ~4-5h spread em 1 dia. Bisect-ability preservada via 3 commits atômicos.
**Source:** Comparação com tracking de "lições aplicadas pre-execute" em CONTEXT.md §10 + matriz UAT consolidada
