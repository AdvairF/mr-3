## Bug pré-existente `_so_custas` em ModuloDividas (NÃO regressão da 7.10bcd)

Durante UAT cross-check inicial, ~7 linhas com `valor_original R$ 0,00` ficaram travadas em "Calculando..." indefinidamente em `TabelaDividas` LOCAL. Hipótese inicial do usuário: Set lookup novo da 7.10c retornaria `[]` para dívidas-fantasma `_so_custas:true` (custas avulsas Phase 7.9), motor travaria.

**Investigação confirmou**: bug é **pré-existente desde Phase 7.9** (introduction de custas avulsas), **não regressão da 7.10bcd**:

- Motor `calcularSaldosPorDivida` (`devedorCalc.js` L172-176) **filtra `_so_custas` antes de calcular**:
  ```js
  const dividasCalc = parseDividas(devedor?.dividas).filter(
    (d) => !d._nominal && !d._so_custas
  );
  if (!dividasCalc.length) return {};
  ```
- Para dívida `_so_custas`, `result[String(div.id)]` nunca é setado → `saldosMap[id_so_custas] === undefined`
- `TabelaDividas.jsx` L94-96 (linha **NÃO tocada por 7.10c**, intacta desde commit `f663e20` CR-05):
  ```js
  const saldo = saldosMap != null ? (saldosMap[String(d.id)] ?? null) : null;
  const saldoExibido = d.saldo_quitado === true ? 0 : saldo;
  ```
- `saldoExibido === null` → renderiza "Calculando..." (label enganoso — não está calculando, é leak de `undefined`)
- Pre-7.10bcd, comportamento idêntico: motor filtrava `_so_custas` desde 7.9 independente da fonte de pagamentos

**Confirmação cross-check**: usuário verificou em PROD (7.10a) que linhas com `valor_original R$ 0,00` mostram **mesmo "Calculando..." indefinido**. Comportamento idêntico em PROD e LOCAL → bug pré-existente confirmado.

**D-34 atomicidade preservada**: rollback NÃO disparou. Phase 7.10bcd é semanticamente correta — UAT continuou com dataset apropriado (dívidas regulares, valor_original > R$ 0).

Escopo: tratar `_so_custas:true` em TabelaDividas/DetalheDivida — **não fix trivial 1 linha** (decisão arquitetural pendente entre helper compartilhado `calcularValorAtualizadoCustasAvulsas` vs inline em cada componente, com correção INPC desde data da custa).
