# Phase 7.9 Learnings: custas-judiciais-crud

## Decisions

### Q1 ambos → avulsas-only (RECONSIDERED)
Custas pertencem ao contrato/processo, NUNCA a documento individual. Q1 original era (c) "ambos vinculadas+avulsas"; após implementar (commit c89a183, +692/-13) e descobrir em UAT-time que prática forense real é só avulsas, foi feito rollback completo + re-plan como (b) avulsas-only.

**Rationale:** "Flexibilidade" foi argumento abstrato; prática forense concreta é só avulsas (custas iniciais, perícia, honorários sucumbenciais são do processo como um todo, não de NF/Cheque específico).
**Source:** 07.9-CONTEXT.md (Q1 RECONSIDERED), 07.9-01-PLAN.md (open_questions Q1 narrativa de rollback)

### D-01 estrito mantido (motor Art.354 intocado)
Motor `devedorCalc.js + services/pagamentos.js + services/dividas.js` ZERO touch. Após Q1 simplificar pra avulsas-only, não há rename canônico necessário (custa shape usa `data` legacy que motor já lê em L484).

**Rationale:** Schema atual via `_so_custas:true + custas:JSONB` modela "custa do contrato" sem precisar tocar motor. Single source of truth preservado sem alias dual.
**Source:** 07.9-CONTEXT.md §D-01 invariante, 07.9-01-PLAN.md Gates C1+C2

### D-22 integral only (toggle pago, sem tabela pagamentos_custa)
Custa tem flag `pago: boolean` + `data_pagamento: "YYYY-MM-DD"`. Zero pagamento parcial. Zero tabela separada. Toggle on/off via service.

**Rationale:** Manutenibilidade > flexibilidade. Custas são integrais por natureza (advogado paga R$X de uma vez, não em parcelas).
**Source:** 07.9-CONTEXT.md §D-22

### D-23 UI simplificada (1 botão, 3 campos, sem dropdown)
NovaCustaModal: 3 campos (Descrição, Valor, Data do pagamento) + 2 botões. ZERO dropdown de vínculo a documento.

**Rationale:** Q1 RECONSIDERED tornou dropdown sem propósito (sem path vinculada). Modal simplificado reflete prática real.
**Source:** 07.9-CONTEXT.md §D-23, NovaCustaModal.jsx

### D-24 evolved: 1 linha única (era split pagas/em-aberto)
DecomposicaoSaldoModal renderiza apenas "Custas pagas atualizadas" = motor.custas.atualizado. A versão original (split pagas/em-aberto) foi descartada após decisão UX P2 (custas sempre nascem pagas) + P3 (correção sempre aplicada independente de pago).

**Rationale:** D-24 split antigo codificava pré-suposição de fluxo "criar em aberto → pagar depois" que não acontece na prática real.
**Source:** 07.9-01-SUMMARY.md (SC1 7-commit chain commits 3d8f80f + 66e5c14)

### D-25 indexador herdado
Custa não tem campo `indice_custa`. Motor usa indexador da dívida-pai (regular = `divida.indexador !== "nenhum" ? indexador : "inpc"`; avulsa `_so_custas:true` = hardcoded "inpc" L485).

**Rationale:** Custa não tem vida própria de correção. Mudar indexador do contrato → todas custas em aberto recorrigem.
**Source:** 07.9-CONTEXT.md §D-25

### Schema unchanged — `_so_custas` JSONB mantido
ZERO migration nova. Custas avulsas ficam em dívida-fantasma (`_so_custas:true`, `valor_total:0`, `custas:[...]`). Tabela `dividas` já tinha colunas `_so_custas BOOLEAN + custas JSONB` desde migration 002.

**Rationale:** Schema atual já modela "custa do contrato". Tabela própria `custas_contrato` exigiria migration + motor edit + cache redesign — phase ficaria 3x maior. Modelo torto mas funcional.
**Source:** 07.9-CONTEXT.md §10 Notas, conversa de diagnóstico Q1 reversão

### Custas SEMPRE nascem pagas (UX decisão P2)
NovaCustaModal envia `pago:true + data_pagamento=form.data`. Botão "Pagar" removido da tabela. Botão "Despagar" também removido (P4). Service `togglePagoCusta` preservado em código mas não exposto na UI.

**Rationale:** Prática real: advogado paga custa de antemão (custas iniciais ao protocolar, perícia ao agendar). Reembolso vem do devedor com correção desde o pagamento.
**Source:** 07.9-01-SUMMARY.md (SC1 commits 3d8f80f + 66e5c14)

### Helper-first: usar buscarContrato existente
criarCusta usa helper `buscarContrato(contratoId)` (L91-93 já exportado em contratos.js) ao invés de duplicar `dbGet("contratos", ...)` inline.

**Rationale:** DRY + zero risco de errar string mágica de tabela. O fix anterior (commit 0c040d4) duplicou dbGet inline e bateu em tabela errada (`contratos` legado em vez de `contratos_dividas` v1.4) — bug fixado em fbbb5fc.
**Source:** 07.9-01-SUMMARY.md (SC1 commit fbbb5fc), feedback_db_integration_gate_missing.md

### 4 callsites do saldo alinhados em 1 fórmula
`motor.saldoAtualizado + motor.custas.atualizado` consistente em: Resumo Financeiro card (DetalheContrato L582), DecomposicaoSaldoModal "Saldo Devedor Atualizado" (L62), useSaldoAtualizadoCache cache (L66 — alimenta listagem Contratos), tabela custas (custasUnificadas useMemo).

**Rationale:** Motor expõe `custas.atualizado` lado-a-lado de `saldoAtualizado` mas não agrega. UI soma client-side (D-01 strict — motor untouched).
**Source:** 07.9-01-SUMMARY.md (SC1 commits 3d8f80f + d354daa)

## Lessons

### Q1 (c) ambos foi over-engineered — UAT-time discovery
Implementação completa shipou e foi rolled back antes do push. Custo: ~1 round completo de planning + implementação (692 linhas, 8 arquivos, 23/23 tests).

**Context:** Lição registrada em `memory/feedback_discuss_walkthrough_fluxo_real.md`: durante /gsd-discuss-phase, exigir walkthrough mental do fluxo real ANTES de lockear escopo de feature com múltiplas opções (a/b/ambos). 3 perguntas obrigatórias: "qual escolheria primeiro?", "alguma você NÃO usaria nunca?", "alguém no workflow usaria diferente?". Se 2+ red flags → simplifica pra single option.
**Source:** 07.9-01-PLAN.md open_questions Q1, feedback_discuss_walkthrough_fluxo_real.md

### Gates Plan 01 NÃO validam DB integration nem semantic drift
Build OK + tests vitest + grep estático passaram com 23/23 (Round 1) e 20/20 (Round 6). UAT humano pegou 4 bugs em série em ~30min: NULL violation FK, tabela errada, UI bloqueava correção, saldo divergente em 3 callsites.

**Context:** Lição central registrada em `memory/feedback_db_integration_gate_missing.md`. Protocolo proposto: schema completeness check + helper-first audit + single-source-of-truth check no PLAN. D-26 light (sem custo de test env) viável imediatamente.
**Source:** 07.9-01-SUMMARY.md SC1 7-commit chain, feedback_db_integration_gate_missing.md

### Gates não detectam tabela errada — falha silenciosa
`dbGet("contratos", ...)` em projeto que tem `contratos` (legado filaDevedor) E `contratos_dividas` (v1.4) NÃO dá erro de tabela inexistente. Retorna array vazio. Lookup falha silenciosamente até virar throw na próxima validação ("contrato não encontrado").

**Context:** Bug 2 da cadeia 7.9 SC1. Causa raiz: assumi que `adicionarDocumento L269 fazia lookup` sem ler a function declaration (recebe `contrato` como parâmetro do caller). Citar `L269 de adicionarDocumento` sem ler XX-30 a XX+30 foi como o bug passou.
**Source:** 07.9-01-SUMMARY.md commit fbbb5fc

### UI codificou bug antigo como contrato em test
Shield 18 (`cache.saldo === motor.saldoAtualizado raw`) codificou pré-fix-P7 como contrato esperado. Após P7 fixar 3 callsites pra somar custas, Shield 18 quebrou — falha foi sinal de evolução semântica, não regressão de comportamento. Test atualizado com comentário forense documentando motivo.

**Context:** Test que assertava "X é Y" pode estar codificando bug que UI tinha. Antes de "ajustar teste", revisar se contrato antigo era certo OU se test codificou bug.
**Source:** 07.9-01-SUMMARY.md commit d354daa

### Semantic drift entre callsites do mesmo valor
3 lugares lendo motor.saldoAtualizado pra exibir "Saldo Atualizado" (Resumo card, Decomposição modal, listagem cache) divergiam após fix pontual em 1 deles. Bug 4 da cadeia SC1 (commit d354daa). Fix manual em 1 callsite deixa outros 2 stale silenciosamente.

**Context:** Single-source-of-truth check pra valor derivado em múltiplos callsites: listar todos no PLAN, garantir fórmula idêntica ou helper compartilhado. Quando muda 1, audit obrigatório dos outros.
**Source:** 07.9-01-SUMMARY.md commits 3d8f80f + d354daa, feedback_db_integration_gate_missing.md

### Pattern claim em commit message DEVE citar linha E ter sido lido inteiro
Commit 0c040d4 message dizia "Pattern: adicionarDocumento L269". Eu vi `devedor_id: contrato.devedor_id` no L269 (body) mas não notei que `contrato` era parâmetro da function (L256), não lookup interno. Erro de leitura superficial → fix duplicou dbGet inline com tabela errada.

**Context:** Citar `LXX de funcaoY` sem ler XX-30 a XX+30 (function declaration + body) é red flag. Pattern claims devem ser auditáveis.
**Source:** 07.9-01-SUMMARY.md commits 0c040d4 + fbbb5fc, feedback_db_integration_gate_missing.md

### Forense de teste atualizado deve documentar motivo
Quando assertion antiga é trocada porque codificava bug, comentário forense obrigatório no teste documentando: contrato novo + commit de mudança + motivo da assertion anterior estar errada.

**Context:** Shield 18 atualizado em commit d354daa inclui 5 linhas de comentário explicando que cache.saldo agora soma custas (versão anterior assertava === motor.saldoAtualizado raw, codificando bug onde listagem mostrava saldo SEM custas).
**Source:** 07.9-01-PLAN.md (commit d354daa) — Shield 18 trecho atualizado

## Patterns

### Fix-forward chain (NOT amend) preserva audit trail
Bugs descobertos em UAT geram NOVO commit fix(...) ao invés de amend ao commit base. 7 commits da phase 7.9 (1 feat + 6 fix-forward) documentam cada bug + causa raiz + solução individualmente.

**When to use:** Bugs descobertos pós-commit base (gates verdes), antes de push. Cada commit fix-forward é auditável e atomicamente revertível. Evita reescrever histórico que já passou em gates.
**Source:** 07.9-01-SUMMARY.md (SC1 7-commit chain), feedback_db_integration_gate_missing.md

### PAUSAs blocking entre plan tasks (UAT + diff review + bump review)
Plan 02 estrutura: Task 1 PAUSA #1 (UAT) → Task 2 PAUSA #2 (diff submódulo pós-push) → Task 3 (bump pai) → Task 4 PAUSA #3 (cross-check SHA) → Task 5 (SUMMARY + docs commit). Cada PAUSA é `checkpoint:human-verify gate="blocking"`.

**When to use:** Phases que terminam em push prod. PAUSAs evitam push de bug + bump com SHA errado + tag prematura. Custo: tempo humano. Benefício: ZERO produção quebrada (bugs ficam pre-push).
**Source:** 07.9-02-PLAN.md tasks block

### Memory feedback indexed em MEMORY.md por phase
Cada lição que generaliza além da phase atual vira arquivo `memory/feedback_<topic>.md` com YAML frontmatter (name, description, type) + linha em `MEMORY.md` (índice de uma linha por entry).

**When to use:** Lição que se aplica a próximas phases (regra/protocolo). 3 novos da 7.9: discuss_walkthrough_fluxo_real, db_integration_gate_missing, helper-first (consolidado no db_integration). Total 13 entries indexadas.
**Source:** memory/MEMORY.md, feedback_*.md files

### dbGet com PostgREST query string (não objeto)
API real do `dbGet(table, queryStr)` é `dbGet("dividas", "id=eq.${uuid}&limit=1")`. PLAN.md usou sintaxe de objeto fictícia → corrigido em deviation auto-fix do executor.

**When to use:** Lookups em Supabase via dbGet. Sempre query string PostgREST, encodeURIComponent pra UUIDs.
**Source:** 07.9-01-SUMMARY.md Deviations §1 + commit c60c38a

### Helper buscarX/listarX existente antes de duplicar dbGet inline
Antes de adicionar dbGet inline em service novo, `grep -n "^export (function|async function)" src/services/<modulo>.js`. Se buscarContrato/listarDocumentos/etc existir e cobrir caso, USAR.

**When to use:** Phase nova adicionando função em service existente que precisa lookup em mesma tabela que já tem helper.
**Source:** 07.9-01-SUMMARY.md commit fbbb5fc, feedback_db_integration_gate_missing.md

### UI computa derived value client-side (D-01 strict)
Quando motor expõe valores parciais lado-a-lado mas não agrega (ex: `saldoAtualizado` + `custas.atualizado` separados), UI soma client-side em vez de tocar motor. Mantém D-01 strict (motor zero edit) com fórmula auditável em código UI.

**When to use:** Motor agrega total mas não decomposição per-item, OU expõe peças que a UI precisa compor diferente. Helper compartilhado se 5+ callsites; fórmula inline replicada se 2-3 callsites com testes.
**Source:** 07.9-01-SUMMARY.md (commits 504e83e + 3d8f80f + d354daa), DetalheContrato L582 + DecomposicaoSaldoModal L62 + useSaldoAtualizadoCache L66

### Comentário forense em test atualizado documenta motivo
Test atualizado pós-evolução de contrato inclui ~3-5 linhas de comentário explicando: contrato novo + commit que mudou + por que assertion anterior estava errada.

**When to use:** Test que codificava bug é corrigido. Sem o comentário, futuro reader pode reverter test pensando que é regressão.
**Source:** 07.9-01-SUMMARY.md commit d354daa, custasCache.test.js Shield 18

## Surprises

### UAT humano pegou 4 bugs em ~30min que gates Plan 01 não pegaram
Build + 20/20 tests + grep estático verdes em commit c60c38a. SC1 single-feature UAT em localhost expôs: NULL FK violation, tabela errada, UI bloqueava correção (lógica D-24 antiga), saldo divergente em 3 callsites. Cada bug exigiu novo round diagnose+approve+fix-forward.

**Impact:** 6 commits fix-forward (0c040d4 → d354daa) + 1 ajuste de Shield 18. Phase shippada com confiança ALTA mas custo de iteração ~6× vs phases anteriores. Lição central da phase, registrada em memory.
**Source:** 07.9-01-SUMMARY.md (SC1 7-commit chain), feedback_db_integration_gate_missing.md

### Rollback completo de commit c89a183 antes do push (zero pollution prod)
Q1 (c) ambos+vinculada implementação shipou submódulo (commit c89a183, +692/-13) e foi 100% rolled back via `git reset --hard HEAD~1` antes de qualquer push origin. Discovery foi UAT-time mental (usuário percebeu que prática real é só avulsas), não bug.

**Impact:** ~1 round completo de planning + implementação descartado. Sem custo prod, sem rebase complicado, sem cleanup. Mas custo humano não-trivial (~3-4h re-planning + re-execute). Vira lição em memory.
**Source:** 07.9-CONTEXT.md (Q1 RECONSIDERED 2026-04-25), 07.9-01-PLAN.md open_questions Q1

### Motor não trata `_so_custas:true` igual às regulares
devedorCalc L478-491 itera `dividasSoCustas` em loop separado, agrega `totalCustasOriginal/Atualizado` mas NÃO push entry em `detalhePorDivida[]`. UI custasUnificadas inicialmente buscava no detalhePorDivida → undefined pra avulsas → fallback nominal (zero correção visível).

**Impact:** Bug 3 da cadeia SC1 (commit 504e83e). Fix: UI calcula correção per-custa client-side via calcularFatorCorrecao import direto.
**Source:** 07.9-01-SUMMARY.md commit 504e83e, devedorCalc.js L478-491

### Existem 2 tabelas `contratos` no projeto
`contratos` (legado, usado por filaDevedor.js) coexiste com `contratos_dividas` (v1.4 atual, usado por contratos.js). Confundir uma pela outra é silencioso — query retorna [] sem erro de tabela inexistente.

**Impact:** Bug 2 da cadeia SC1 (commit fbbb5fc). Lição registrada em memory: cuidado redobrado em phases que tocam services num projeto com múltiplas tabelas de prefixo similar.
**Source:** filaDevedor.js + contratos.js TABLE constants, feedback_db_integration_gate_missing.md

### dbGet API real difere do PLAN snippet
PLAN.md usou `dbGet("dividas", { id: 'eq.${uuid}' })` (sintaxe objeto). API real é PostgREST query string `dbGet("dividas", `id=eq.${encodeURIComponent(uuid)}&limit=1`)`. Auto-fixed pelo executor com Rule 1 deviation.

**Impact:** Build/test passaram porque editarCusta/excluirCusta/togglePagoCusta seriam chamados só em contexto runtime. PLAN não conferiu API real antes de citar. Lesson: protótipos de service novos devem ser smoke-testados contra dbGet real.
**Source:** 07.9-01-SUMMARY.md Deviations §1

### Btn componente não passa `type` prop através
`<Btn type="submit">` dentro de `<form onSubmit>` resultaria em Cancelar acionando submit (porque type não passa pro `<button>` underlying). Auto-fixed: removido `<form>`, usado `<div>` + onClick direto no Btn Salvar.

**Impact:** UX bug evitado pré-UAT. Lesson: assumir API de componente UI precisa leitura do código do componente; não inferir de "react idiomatic".
**Source:** 07.9-01-SUMMARY.md Deviations §2

### Motor hardcoda "inpc" pra `_so_custas:true` (L485) — Shield 19 fixture errada
Shield 19 (indexador herdado) original usava dívida `_so_custas:true` + indexador "IPCA" uppercase. Motor L485 ignora `divida.indexador` quando `_so_custas:true` (hardcode "inpc"); calcularFatorCorrecao keys são lowercase. Fixture trocou pra dívida regular + lowercase indexador.

**Impact:** Test inicial falsamente passou (mesmo fator computado) ou falhou (uppercase retorna fator=1). Ajustado pré-commit. Documenta divisão entre regular dividas (D-25 herança) e _so_custas:true (hardcoded "inpc" L485).
**Source:** 07.9-01-SUMMARY.md Deviations §3, custasCache.test.js Shield 19

### Parent + submódulo compartilham mesmo URL GitHub (master vs main)
`https://github.com/AdvairF/mr-3.git` é remote tanto do parent (branch `master`) quanto do submodule (branch `main`). Setup inusitado mas válido — push origin master vai pra mesmo repo que push origin main do submódulo, em branches diferentes.

**Impact:** Push reports do parent mostraram URL "mr-3" causando momento de "tabela errada de novo?". Confirmado via git remote -v que é o setup do projeto. Sem ação corretiva necessária.
**Source:** Plan 02 push pai output + git remote -v
